现在来看方向 B vs C 的对比：

## 方向 B：让 `_budget_adjust` 用满 prefer_level 粒度

当前 prefer_level 有 5 档（0.0/0.2/0.5/0.7/0.9），但 `_budget_adjust` 只认两个阈值（≥0.8 / ≤0.3），中间三档形同虚设。

```
prefer_level 当前映射：
  0.9 = high（浪费 >30%） → 全部推 qwen
  0.7 = moderate（浪费 10-30%） → 不做任何调整
  0.5 = low（浪费 5-10%） → 不做任何调整
  0.2 = depleted（< 5%） → 不做任何调整
  0.0 = exhausted/expired → 全部推 deepseek
```

**B 的做法：** 给每个档位配不同的路由策略，精细调控，比如：

| prefer | 策略 |
|--------|------|
| ≥0.8 | 全部 moderate+complex → qwen3.7-max |
| 0.6-0.7 | 仅 moderate+low/medium → qwen3.7-max |
| 0.4-0.5 | 仅 simple+low → qwen3.6-plus |
| 0.2-0.3 | 不调整（L2 默认） |
| ≤0.1 | 全部回 deepseek |

**优点：** 理论上浪费/耗尽之间过渡更平滑，不会跳变。

**缺点：** 策略表复杂了，但收益很小——中间档位（0.5/0.7）持续的时间很短（几天就跨过去了），不值得为过渡态写四条规则。而且 `prefer_level` 每天才更新一次，太细的档位来不及反应。

## 方向 C：控制回路——按剩余/天数定目标消耗

撤销投影预测，改为：

```
target_daily_burn = credits_remaining / days_left
```

然后 `prefer_level = clamp(target_daily_burn / normal_burn, 0.0, 1.0)`

如果目标消耗是正常消耗的 2 倍，prefer_level = 1.0；如果目标消耗低于正常消耗，prefer_level = 0.0。

**优点：** 没有反馈回路问题——prefer_level 直接驱动消耗，消耗结果反向影响下一个 target，但 target 是根据剩余硬算的，不受消耗速率影响。

```
当前场景：18,604 / 20 天 = 930/天
当前正常消耗（不走 Token Plan 时）：~1,230/天
930 / 1230 = 0.76 → prefer_level ≈ 0.76 → 足够让 moderate 走 qwen
```

**缺点：** 需要额外参数（"normal burn"——全量走 Token Plan 时的基准消耗），得从历史数据估计或分层统计。

## 结论对比

| 维度 | 方向A（已实施） | 方向B（粒度填充） | 方向C（控制回路） |
|------|----------------|------------------|-----------------|
| 改动量 | 3 行 + 2 处 | ~10 行配置表 | ~30 行新算法 |
| 反馈回路风险 | 已消除（budget_context 同步修了） | 低，但中间档位没用 | 无（开环控制） |
| 过渡平滑性 | ~7 天后跳变到 0.0 | 多个台阶过渡 | 连续平滑 |
| 新增复杂度 | 几乎无 | 中（策略表维护成本） | 中（需基准消耗参数） |
| 实际收益 | **最大**（解决核心矛盾） | 低（中间态时间短） | 中（过渡期更优） |

**我的建议：方向A够了，方向C可以作为下一个迭代。** 方向B不值得——它的中间档位持续时间太短（每次 prefer_level 跨档只维持几天），为几天的过渡写四条规则纯属增加维护成本。

方向C可以等 Token Plan 下一周期（7月续费后）再考虑——到时候配新的初始参数会更准确。要现在同时做方向C吗？
