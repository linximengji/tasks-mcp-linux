现在完全清楚了。来看核心问题：

## prefer_level 的反馈回路缺陷

`balance_query.py` 的计算逻辑：

```python
projected_remaining = credits_remaining - daily_burn * days_left
# daily_burn = 过去 7 天的平均消耗
if projected_remaining <= 0:
    prefer_level = 0.0  # "耗尽" → 禁止用 Token Plan
```

`model_proxy.py` 的使用逻辑：

```python
if prefer_level <= 0.3:
    # moderate+complex 路由改走 deepseek-v4-pro（按量计费）
```

**这里有一个循环依赖：**

```
prefer_level = 0.0 → 避开 qwen 模型 → Token Plan 日消耗下降
                                     → 但 daily_burn 是 7 天均值
                                     → 数值还挂在高位（1,232/天）
                                     → 预测 still 显示 "会耗尽"
                                     → prefer_level 继续保持 0.0
```

实际上，在 prefer_level = 0.0 生效后，qwen 流量只有 ~1.2%（被 budget_adjust 压制的），真实的 Token Plan 消耗远低于 1,232/天。**Prediction 是错的，系统在自欺欺人。**

## 真正矛盾：exhausted = 0.0 的逻辑是反的

当前策略解释：exhausted → "省着用" → prefer_level = 0.0

但 Token Plan 积分是**沉没成本、用罄即弃**的。当预测会耗尽时，正确的行为应该是**加快用掉**——每个走 qwen（Token Plan）的请求都是免费的，等于帮按量计费省钱。

| 场景 | 当前 prefer_level | 合理行为 |
|------|-------------------|---------|
| 预测盈余 >30% (waste high) | 0.9 ✅ | 多用 TP |
| 预测盈余 10-30% | 0.7 ✅ | 偏好 TP |
| 预测盈余 5-10% | 0.5 ⚠️ | 均衡 |
| 预测刚好耗尽 | **0.0 ❌** | 应该提高到 ~0.8，加快消耗 |
| 已过期 | 0.0 ✅ | 没办法了 |

## 第二个问题：_budget_adjust 的粒度浪费

prefer_level 有 5 档（0.0, 0.2, 0.5, 0.7, 0.9），但 `_budget_adjust` 只认两个阈值：

```
≥ 0.8 → 推给 qwen
≤ 0.3 → 推给 deepseek
0.4-0.7 → 不做任何调整（全靠 L2 默认路由）
```

0.5、0.7 两档形同虚设。而且 ≥0.8 时也只处理了 `moderate+low → qwen3.7-max` 和 `simple+low/medium → qwen3.6-plus`，`moderate+medium/high` 和 `complex` 这些本该优先喂给 Token Plan 的却漏了。

## 建议的调整方向

**方向 A（最小改动）：修复 exhausted 逻辑**

`balance_query.py` 里只改一个数值：

```python
# 当前
projected_remaining <= 0 → prefer_level = 0.0
# 建议
projected_remaining <= 0 → prefer_level = 0.8  # 趁还有额度加快消耗
```

这样当预测会耗尽时，_budget_adjust 走 ≥0.8 分支，往 qwen 推。现有 ≥0.8 的规则已经正确，不需要改 `model_proxy.py`。

**方向 B（加点料）：让 `_budget_adjust` 用满 prefer_level 粒度**

在 model_proxy.py 里补全 ≥0.8 时缺失的规则，加上中间档位：

```python
# ≥ 0.8 时补充：
complex 任意 → qwen3.7-max
moderate medium/high → qwen3.7-max

# 0.4-0.6 时：
simple/low → qwen3.6-plus

# 0.6-0.8 时：
moderate low → qwen3.7-max
```

**方向 C（重做机制）改成控制回路**

不投影预测，而是每天算：`目标消耗 = 剩余积分 / 剩余天数`，然后让 `prefer_level` 追赶这个目标。避免反馈回路问题。
