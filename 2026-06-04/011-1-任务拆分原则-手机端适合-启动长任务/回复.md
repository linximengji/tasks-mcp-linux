# Claudetalk 手机端优化方案

先对照现状盘点已有能力，再补空白。

## 现状盘点

| 维度 | 已有 | 缺口 |
|------|------|------|
| **入口** | 飞书私聊 + 群聊双通道，peer message 差异已处理 | 无法区分"手机端" vs "桌面端飞书" |
| **长任务** | phone-tasks-archive 归档私聊对话 | 无启动长任务 + 完成通知链路 |
| **审批** | Bash hooks 拦截长命令，PermissionRequest 过滤 | 无飞书端审批流，必须回到桌面 |
| **会话** | x-claude-session-id header 支持 | 无 /rename、/session 切换命令 |
| **输出** | MCP SSE server 已建 | 未针对手机屏优化（长代码块、diff） |
| **成本** | proxy 路由 + token-stats 日报 | 无手机端快速查余额命令 |

## 优化方案（按投入产出排序）

### P0：低成本高回报（2-3 天）

**1. 手机/桌面信号识别**

飞书 webhook 事件带 `app_id` + 用户端信息。最简方案：私聊消息如果来自飞书移动端（user_agent 含 Mobile），注入 `is_mobile=true` 标记到 system prompt。

```typescript
// claudetalk/src/feishu/handler.ts
if (event.source === "mobile") {
  context.isMobile = true;
  context.maxOutputLength = 800;  // 手机屏一屏能看完
}
```

**2. 手机模式下的输出适配 hook**

在 PreToolUse 或 PostToolUse 加一个 hook：当 `is_mobile=true` 时，把工具输出截断到 800 字符，长内容写临时文件，只给一行摘要 + 文件路径。

```python
# ~/.claude/hooks/mobile_trim.py
import os, json
if os.environ.get("IS_MOBILE") == "1":
    content = json.loads(sys.argv[1])
    if len(content) > 800:
        path = f"/tmp/claude_{uuid4()[:8]}.txt"
        with open(path, "w") as f: f.write(content)
        print(f"[内容过长，已写入 {path}]")
```

**3. 飞书快捷命令集**

复用现有 MCP 通道，加几个快捷命令：

| 命令 | 作用 | 场景 |
|------|------|------|
| `/token` | 查当前 Token Plan 余额 | 随手看一眼 |
| `/session` | 列出活跃会话 + 切换 | 手机接续桌面任务 |
| `/approve` | 批准/拒绝待审批操作 | 不在电脑前时授权 |
| `/task <描述>` | 启动长任务，完成后飞书通知 | 跑测试、生成报告 |
| `/diff <PR>` | 显示 PR 摘要 + 文件列表 | 手机 code review |

### P1：中等投入，补关键缺口（1 周）

**4. 飞书审批流**

Hook 检测到高风险操作时（匹配正则：`rm -rf`、`git push --force`、生产环境相关），不在 CC 内等待确认，而是：

1. CC 暂停等待（HTTP polling）
2. Claudetalk 发送飞书卡片消息：`⚠️ 等待批准：rm -rf /tmp/cache`
3. 卡片带两个按钮：**批准** / **拒绝** / **查看详情**
4. 用户点按钮 → CC 收到回调继续或中止
5. 5 分钟无响应 → 自动拒绝 + 飞书通知

```python
# hook 伪代码
HIGH_RISK = re.compile(r'rm\s+-rf|git push --force|kubectl delete|DROP TABLE|production')
if HIGH_RISK.search(command) and context.is_mobile:
    approval_id = feishu.send_approval_card(command)
    decision = wait_for_approval(approval_id, timeout=300)
    return decision  # "approved" / "rejected" / "timeout"
```

**5. 任务生命周期管理**

手机端 `/task 跑一遍 proxy 回归测试` 触发的流程：

```
1. CC 接受任务，立刻回飞书：「任务已启动 #T-042，预计 3 分钟」
2. 后台执行，每 30s 发一次进度：「3/12 tests passed...」
3. 完成发总结：「✅ 12/12 通过，详情见 [链接]」
4. 失败发详情：「❌ Test 7 failed: L2 classifier timeout」
```

实现要点：
- 用 `task_id` 关联飞书消息 + CC session + 输出文件
- 进度报告通过 `streaming` SSE 事件触发，不是轮询
- 完成后把完整输出存到 `~/.claude/task-archive/T-042.md`

**6. 会话快速切换**

`x-claude-session-id` header 已支持。补一个命令层：

```
/session list     → 列出最近 5 个会话（首条消息 + 最后活动时间）
/session <id>     → 切换到该会话
/session rename <id> <name>  → 重命名
/session new      → 新建
```

会话列表缓存在 `~/.claude/sessions.json`，claudetalk 启动时加载。

### P2：锦上添花（按需）

**7. Code review 紧凑模式**

手机端 review PR 时，把 diff 折叠成：

```
📦 auth.py +42 -8
  L23: 新增 OAuth flow（需确认 refresh token 处理）
  L89: ⚠️ 硬编码 secret（建议用 env）

📦 utils.py +5 -0
  L12: 纯格式化，无问题
```

不要发完整 diff。Claude 自己总结问题清单，文件级粒度。

**8. 自动上下文切换提示**

检测到用户从桌面切到手机（同 session 但 user_agent 变化），主动提示：

```
🔄 检测到切换到手机端。上一轮讨论：proxy routing 改造，已完成 3/5 步骤。
要继续第 4 步吗？回复 1 继续 / 2 换话题
```

## 不建议做的

| 提案 | 为什么跳过 |
|------|----------|
| 复杂多文件重构放手机 | 认知负担太大，不适合触屏 |
| Git worktree 管理 | 个人开发者单分支足够，worktree 是团队场景 |
| 权限分级到 5 档 | 当前 bash hook 的 2 档（放行/拦截）已够 |
| 会话自动归档 + AI 摘要 | 成本高，手机场景用不到 |

## 实施优先级建议

```
Week 1: P0 三项（信号识别 + 输出适配 + 快捷命令）
        ↓
Week 2: P1-4（飞书审批流）
        ↓
Week 3: P1-5/6（任务生命周期 + 会话切换）
        ↓
按需：  P2（review 紧凑模式、上下文切换）
```
