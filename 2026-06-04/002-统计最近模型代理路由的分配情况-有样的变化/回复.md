# 模型代理路由分配分析

## 当前分布（6/3 全天数据，2247 次请求）

| 模型 | 占比 | 日均请求 |
|------|------|---------|
| `deepseek-v4-flash` | **49.6%** | ~1100 |
| `deepseek-v4-pro` | **49.2%** | ~1100 |
| `qwen3.7-max` | **1.2%** | ~27 |

## 近期变化 & 效果

**1. L1 "recent tools" 规则移除（6/3）** ✅ 正向效果

以前工具调用后几乎所有请求都被强制路由到 pro（~90% 误判），现在由 L2 正常分类。half-half 的分布说明这个改动达到了预期——flash 扛住了简单追问，pro 处理中等复杂度任务。

**2. 预算策略全面保守化（prefer=0.0）**

`routing_policy.json` 当前判断 Token Plan 必然耗尽（到期预计 -6026），所以策略是全量避险——L2 判定 moderate → 本该走 qwen3.7-max（Token Plan）的，全部拉到 deepseek-v4-pro（按量计费）。L2 本应有 ~8% 的流量路由到 qwen3.7-max，实际只有 1.2%。

## 潜在问题

### 核心矛盾：Token Plan 没吃完，但按量计费在花真钱

Token Plan 剩余 **18,604 积分**，20 天后到期。日常消耗 **1,232/天**，按当前走法到期还剩 **6,027 积分没用完**——已经买好的额度浪费了。

同时 moderate 级别的请求（~1100/天）走 deepseek-v4-pro 按量计费：
- pro 输入 3 CNY/百万 tokens，qwen3.7-max 输入 10 CNY/百万
- 但 qwen3.7-max 走的是**已购 Token Plan**，边际成本为零
- 真正的好模型放着不用，反而花钱用次一级的 pro

**要不要调整预算策略？** 两个方向：

| 方向 | 效果 | 风险 |
|------|------|------|
| **现状（保守）** | 保护 Token Plan，但到期浪费~6000 积分 | 按量计费成本偏高 |
| **激进（提高 prefer_level）** | 多走 qwen3.7-max，吃满 Token Plan | 消耗完额度后全转按量计费 |

### 其他

- **0 错误、0 警告、1 次 fallback**——系统稳定性没问题
- **6/3 请求量翻倍**（2247），但没触发异常，系统扛住了
- L2 分类器 ~50 tokens/次，日均 ~55K tokens 开销（~0.55/天），可以忽略

## 结论

**系统本身工作良好，没有 bug 需要修。** 唯一值得做的是**预算策略微调**——既然 Token Plan 肯定吃不完，不如把 `prefer_level` 调到 ~0.5，让 L2 判定的 complex 和部分 moderate 走 qwen3.7-max（用剩余额度），减少按量计费支出。要不要调？
