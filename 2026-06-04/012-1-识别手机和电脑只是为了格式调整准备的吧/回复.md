调查结果比预想的严重——**所有来自飞书的命令全部完全绕过授权**。

## 核心发现：`--dangerously-skip-permissions` 是硬编码

`claudetalk/src/core/claude.ts:471` 和 `:665` — 每次通过飞书调用 CC，都加了这个参数。结果：

```
飞书消息 → claudetalk → claude -p --dangerously-skip-permissions
                                    ↓
                          CC 权限系统完全跳过
                          Bash/Read/Write/MCP 全部自动执行
                          PermissionRequest hook 永远不会触发
                          Settings.json 里的 allow/deny 列表不生效
                          ...
```

现在知道我为什么没收到过审批提示了——**根本就不会弹**。

## 所以问题不是"怎么加手机审批"，而是"要不要撤掉危险绕过"

现有安全防护三层，当前状态：

| 防护层 | 交互式 CC | 飞书→Claudetalk |
|--------|-----------|----------------|
| CLAUDE.md 规则 | ✅ 约束 AI 行为 | ✅ 同一份规则 |
| settings.json allow/deny | ✅ 生效 | **❌ 被绕过** |
| Bash hooks（长度/元字符） | ✅ PreToolUse 触发 | ⚠️ PreToolUse 触发但无弹窗 |
| PermissionRequest hook | ✅ 过滤 | **❌ 永远不触发** |

实际靠的只有 CLAUDE.md 的 AI 行为约束——没有技术兜底。

## 三个方向

**A：去掉 `--dangerously-skip-permissions`，让 settings.json 的 allowlist 生效**

最直接。settings.json 已经列了 ~50 条全局 allow + ~100 条项目级 allow + deny 列表。去掉 bypass 后：

- ls、git status、WebFetch 等已批准操作 → 自动执行（quiet approve）
- 未在白名单内的命令 → 弹审批（但目前飞书通道无法响应审批）
- **问题**：飞书无法处理 CC 的 stdin 审批提示，会卡死

**B：在 claudetalk 里实现一个权限策略引擎，替代 CC 的权限系统**

不依赖 CC 的 `--dangerously-skip-permissions`，而是在 claudetalk 里自己判断：

```typescript
// 在 callClaudeStreaming 中，不要加 --dangerously-skip-permissions
// 而是在 claudetalk 中维护一份 permissions.json：
{
  "allow":  ["git status", "ls *", "python *", "curl -s *"],
  "deny":   ["rm -rf", "git push --force"],
  "feishu_approval":  ["git push", "npm publish", "Dockerfile修改"]
}
```

- allow → 直接通过
- deny → 拦截+报错
- feishu_approval → 发飞书卡片等用户点批准/拒绝

**C：保持现状，靠 CLAUDE.md 规则约束 + 事后审计**

现状实际没出过问题（CLAUDE.md 规则足够），加审批流程反而增加摩擦。建议先不动。

---

你觉得倾向哪个方向？A 最简单但会让飞书通道失效（所有未在白名单的命令会卡住），B 最完整但需要开发。考虑到你目前通过手机发出的命令基本都是常规操作（git、代码审查、跑测试），方向 C 可能当前就够。
