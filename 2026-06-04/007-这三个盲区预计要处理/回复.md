## 盲区 1：thinking 块 → qwen MAAS

### 问题链

```
① CC 上一轮对话使用了 thinking（DeepSeek 支持）
② 对话历史中残留 thinking/redacted_thinking 块
③ 本轮请求路由到 qwen3.7-max (MAAS)
④ _native_stream_kwargs() 里 body.copy() 直通，不做 sanitize
   → qwen MAAS 收到含 thinking 块的消息
   → 如果 MAAS 端不认识这些块类型，返回 422
```

两个 sanitize 调用点在 `model_proxy.py:203` 和 `:312` 都只判断 `provider == "deepseek"`，`"anthropic"` 路径完全跳过处理。

### 修复方案

在 `sanitize.py` 加一个函数，从 messages 中移除 thinking 块：

```python
ALLOWED_BLOCK_TYPES_NO_THINKING = {
    "text", "tool_use", "tool_result", "tool_reference",
}

def strip_thinking_blocks(body):
    """移除 messages 中的 thinking/redacted_thinking 块。For qwen MAAS 兼容性。"""
    for msg in body.get("messages", []):
        content = msg.get("content")
        if isinstance(content, list):
            msg["content"] = [
                b for b in content
                if not isinstance(b, dict) or b.get("type") not in ("thinking", "redacted_thinking")
            ]
```

然后在 `model_proxy.py` 的 `_route_and_sanitize` 中扩展到支持 "anthropic" provider：

```python
if route and route.get("provider") == "deepseek":
    sanitize.sanitize_for_deepseek(body)
elif route and route.get("provider") == "anthropic":
    sanitize.strip_thinking_blocks(body)  # qwen MAAS 不支持 thinking
```

同样在 `handlers.py` 的 `build_kwargs` 和 `_native_stream_kwargs` 里补上：

```python
if r["provider"] == "deepseek":
    sanitize.sanitize_for_deepseek(upstream)
elif r["provider"] == "anthropic":
    sanitize.strip_thinking_blocks(upstream)
```

**改动量**：sanitize.py +8 行，model_proxy.py +2 行，handlers.py +2 行。

## 盲区 2：streaming 下 tool_call 的 content_block_start 缺 id/name

### 问题代码

`handlers.py` 第 176-182 行：

```python
if tci not in tbs:
    tbs[tci] = {"idx": nbi, "id": "", "name": ""}
    nbi += 1
    yield sse_encode("content_block_start", {        # ← 此时 id/name 还是 ""
        "type": "content_block_start", "index": tbs[tci]["idx"],
        "content_block": {"type": "tool_use", "id": "", "name": "", "input": {}},
    })
tb = tbs[tci]                                        # ← 获取引用在 yield 之后！
if tc.get("id"): tb["id"] = tc["id"]
```

`content_block_start` 发射后，Anthropic 协议**没有**更新 id/name 的 delta 类型。所以客户端收到的 tool_use 块 id=""、name=""。

标准的 OpenAI chunk 结构实际上会在同一个 chunk 里同时包含 id 和 function.name（DeepSeek、大多数 OpenAI 兼容都这样做），所以大多数场景下这行代码不会出问题。但部分向量化较差的 provider 可能分开发送。

### 修复方案

把填充提到 yield 之前：

```python
if tci not in tbs:
    tb = {"idx": nbi, "id": "", "name": ""}
    nbi += 1
    # 先填充当前 chunk 中已有的 id/name
    if tc.get("id"): tb["id"] = tc["id"]
    func_delta = tc.get("function", {})
    if func_delta.get("name"): tb["name"] = func_delta["name"]
    tbs[tci] = tb
    yield sse_encode("content_block_start", {
        "type": "content_block_start", "index": tb["idx"],
        "content_block": {"type": "tool_use", "id": tb["id"], "name": tb["name"], "input": {}},
    })
else:
    tb = tbs[tci]
    if tc.get("id"): tb["id"] = tc["id"]
    func_delta = tc.get("function", {})
    if func_delta.get("name"): tb["name"] = func_delta["name"]
```

**改动量**：handlers.py ~15 行。

## 盲区 3：sanitize 时序

### 结论：没问题，不需要改

sanitize 只在**最终模型确定后**执行，调用点（`model_proxy.py:203`、`:312`、`handlers.py:39-40`、`:344-345`）都检查 `provider == "deepseek"`。不存在"先 sanitize 再切换模型"的路径。

## 汇总

| 盲区 | 优先级 | 方案 | 改动量 | 影响 |
|------|--------|------|--------|------|
| ① thinking→qwen MAAS | **高** | sanitize.py 加 `strip_thinking_blocks`，两处调用 | ~12 行 | 消除潜在 422 |
| ② streaming tool_call id/name | **中** | handlers.py 重排填充顺序 | ~15 行 | 消除空 id 的偶发 bug |
| ③ sanitize 时序 | 无需处理 | — | 0 | 已正确 |

盲区 1 的优先级高——当前 prefer_level=0.8 生效后 moderate 流量会大量转向 qwen3.7-max，thinking 块出现的概率会随之增加。要现在做这两个修复？
