## 问题 1：模型切换时的数据格式对齐

先说结论：**当前路由系统中，主流切换路径格式是兼容的，但有 3 个盲区。**

### 三条格式路径

```
┌─────────────────────────────────────────────────────────────┐
│  CC 发送：Anthropic 格式 (system顶层 + messages[] + tools)   │
└─────────────────────────────────────────────────────────────┘
                              │
           ┌──────────────────┼──────────────────┐
           ▼                  ▼                  ▼
   deepseek-v4-pro     qwen3.7-max(MAAS)   qwen3.7-max-ds
   deepseek-v4-flash   qwen3.6-plus(MAAS)  qwen3.6-plus-ds
   ───────────────     ───────────────     ───────────────
   provider:            provider:           provider:
   "deepseek"           "anthropic"         "openai"
   ───────────────     ───────────────     ───────────────
   格式: passthrough     格式: passthrough    格式: 需转换
   sanitize清洗后直发     不做清洗，直发     anthropic_to_openai()
```

**关键事实：** qwen3.7-max 的 MAAS 端点提供的是 Anthropic 兼容 API（`/apps/anthropic/v1/messages`），所以**三条路径中两条是直通的**，只有 quota_backup 到 DashScope 时需要转换。

### 路由切换的格式兼容性矩阵

| 切换路径 | 格式变化 | 风险 |
|---------|---------|------|
| deepseek ↔ qwen(MAAS) | 无（都是 Anthropic 格式直通） | 🟢 低 |
| deepseek/qwen → qwen-ds | Anthropic → OpenAI 转换 | 🟡 中 |
| qwen(MAAS) ↔ qwen-ds | 直通 ↔ 转换 | 🟡 中 |

### 三个已知盲区

**盲区 1：thinking 块 — 发送到 qwen MAAS 端后可能被拒绝**

CC 可能在 messages 中携带 `thinking` 类型的内容块（来自之前 assistant 的思考输出）。代码路径：

```
_route_and_sanitize()
  → L1/L2 路由到 qwen3.7-max
  → provider = "anthropic"
  → build_kwargs() → body.copy() 直通发送
  → qwen MAAS Anthropic 端点收到含 thinking 块的消息
```

qwen MAAS 端点如果不支持 `thinking` 块类型，会返回错误。但实际测试中这没发生过——CC 的 conversation history 中的 thinking 块来自旧轮 assistant 回复，但 CC 只在确认模型支持 thinking 时才会启用它。由于我们代理了模型名称，CC 可能认为目标是 thinking-capable 模型。

**盲区 2：streaming 下 OpenAI → Anthropic SSE 转换的状态机**

`handlers.py` 中 `handle_anthropic_stream` 的 OpenAI SSE 转换（`else` 分支，第 130-215 行）维护了三个状态变量：

```python
ti = None        # text content_block 的 index
tbs = {}         # tool_calls 的 index 映射表 {openai_index: {idx, id, name}}
nbi = 0          # 下一个 content_block index
```

逻辑是按 OpenAI chunk 顺序推断 Anthropic 的 `content_block_start/delta/stop`。工具调用复杂时有如下顺序问题：

- OpenAI 可能先发 `delta.tool_calls[0].function.name` 再发 `delta.tool_calls[0].function.arguments`
- Anthropic 需要先 `content_block_start`（同时包含 name 和 id）才能发 `input_json_delta`
- 当前代码等 name 和 id 都在 `tbs[tci]` 收集齐了才发 input_json_delta，但 `content_block_start` 在第一次遇到该 tci 时就发了，当时 name/id 可能是空

这在实际中可能看到工具调用的 `content_block_start` 中 name 和 id 为空，然后通过后续 delta 补充。Claude Code 客户端可能对这种"延迟填充"敏感。

**盲区 3：DeepSeek sanitize 在切换时的残留效果**

虽然代码正确地按**最终路由的模型**来决定是否调用 sanitize，但 `_route_and_sanitize()` 的逻辑是：L1 匹配到非 deepseek 模型 → 不 sanitize → 直接返回 → handler 使用。L1/L2 未匹配 → 返回 l2_future → L2 解析后设置 model → 按最终 provider 决定 sanitize。

这条路径是正确的。不会出现"先 sanitize 再切模型"的情况。🟢

### 总结

```
风险等级:
  🟢 thinking块 → qwen MAAS 端点:  未见报错，但理论上(盲区1)
  🟢 streaming tool_call 状态机:  大部分场景正常(盲区2)
  🟢 sanitize 时序:              正确，没有问题(盲区3)
```

目前实际运行中没有格式相关错误报告（0 ERROR），说明这些盲区在当前使用模式下并未触发。

## 问题 2：完整路由逻辑

我用文字流程来描述，结构比图更精确：

### 请求生命周期

```
CC 发送 /v1/messages (Anthropic 格式)
  │
  ├─ 1. model_proxy.py proxy_anthropic()
  │     ├─ set_req_id(uuid)
  │     ├─ embed_images(body)          ← 替换 [图片:path] 为 base64
  │     └─ _route_and_sanitize(body)
  │
  ├─ 2. _route_and_sanitize()
  │     ├─ router.classify(body)       ← L1 四条规则 first-match
  │     │    │
  │     │    ├─ 图片(r1)?       → qwen3.6-plus
  │     │    │                   原因: "L1:image"
  │     │    │
  │     │    ├─ 简短问候(r2)?   → deepseek-v4-flash
  │     │    │   (<400 tokens)   原因: "L1:trivial"
  │     │    │
  │     │    ├─ 极长上下文(r3)? → qwen3.7-max
  │     │    │   (>15000 tokens) 原因: "L1:very-long"
  │     │    │
  │     │    └─ 均未命中        → return None
  │     │                         原因: "l2-classifier"
  │     │
  │     ├─ [L1命中] body["model"]=routed, sanitize(如deepseek), return route
  │     │
  │     └─ [L1未命中] return l2_future (异步分类器请求)
  │
  ├─ 3. [L2] _resolve_classifier(l2_future)
  │     ├─ 请求 deepseek-v4-flash 分类用户消息
  │     │   prompt注入 budget_context:
  │     │     prefer=0.8 → "use credits, prefer qwen (free)"
  │     │     prefer=0.0 → "expired, avoid qwen"
  │     │
  │     ├─ 解析返回值: "complexity budget"
  │     │   ┌──────────┬──────────┬──────────┐
  │     │   │ trivial  │ simple   │ moderate │ complex  │
  │     │   ├──────────┼──────────┼──────────┼──────────┤
  │     │   │ flash    │ flash    │ pro      │ qwen-max │  ← 默认路由
  │     │   └──────────┴──────────┴──────────┴──────────┘
  │     │
  │     └─ _budget_adjust(complexity, budget, prefer_level)
  │          │
  │          ├─ prefer≥0.8: 全部往上提一级
  │          │   simple→qwen3.6-plus, moderate/complex→qwen3.7-max
  │          │
  │          ├─ 0.3<prefer<0.8: 不做调整 (用L2默认)
  │          │
  │          └─ prefer≤0.3: moderate/complex→deepseek-v4-pro
  │
  ├─ 4. 确定最终 route
  │     body["model"] = 最终模型名
  │     route = ROUTES[模型名]
  │     if provider=="deepseek": sanitize_for_deepseek(body)
  │
  ├─ 5a. [非流式] handle_anthropic()
  │     ├─ build_kwargs:
  │     │   deepseek/anthropic: body直通 + 改model/stream字段
  │     │   openai: anthropic_to_openai(body) 转换
  │     ├─ fallback.request_with_fallback()
  │     ├─ [deepseek/anthropic] 直接返回resp.content
  │     └─ [openai] openai_to_anthropic(resp.json()) 转回
  │
  ├─ 5b. [流式] handle_anthropic_stream()
  │     ├─ [deepseek/anthropic] _native_stream_kwargs → SSE passthrough
  │     ├─ [openai] _openai_stream_kwargs → SSE块级转换
  │     │   (OpenAI delta → Anthropic content_block_start/delta/stop)
  │     └─ fallback链: 按 litellm_config.yaml fallback 字段
  │
  └─ 6. telemetry 记录
        ├─ record_request(model, reason)
        ├─ record_tokens(in/out/cache)
        └─ record_latency(ms)
```
