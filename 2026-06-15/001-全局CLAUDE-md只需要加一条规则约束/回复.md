## architecture 节点的合理形态

它不是文件列表，是**功能切面 → 波及范围**的映射。一个项目一个 `architecture.md` 段，结构如下：

```markdown
## 功能切面速查

### message_ingress
消息入口：接收消息 → 分类 → 路由到 Claude
- 入口：src/channels/feishu/index_feishu.ts
- 波及：src/core/classifier.ts, src/core/claude.ts, src/core/session.ts
- 配置：.env (FEISHU_APP_ID, CLAUDE_API_KEY)

### card_action_handling  
飞书卡片按钮回调
- 入口：src/channels/feishu/index_feishu.ts (WSClient)
- 波及：src/core/classifier.ts (fallback), src/mcp-server.ts (methodology)
- 消费者：飞书客户端

### mcp_tools
MCP Server 暴露的工具接口
- 入口：src/mcp-server.ts
- 波及：src/tools/*.ts
- 消费者：claude-code (外部)
```

## 为什么这个粒度合理

- **不存文件列表**：代码即真相，grep/Glob 比记忆更权威
- **存的是"隐含依赖"**：import 能看出来，但 `.env` 配置、外部消费者、跨文件协议约定，import 看不出来，需要记忆
- **变更频率低**：只有新增/删除功能切面或调整入口时才更新，比文件级索引稳定一个数量级
- **体积可控**：中型项目 5-8 个切面，2-3K 字符，`memory_get` 一次读全不心疼

## 触发链

```
SessionStart → 看到 claudetalk 有 architecture 段
    ↓
CLAUDE.md 规则 → "编辑前查 architecture 段"
    ↓

    ↓

```

比 CLAUDE.md 内嵌速查表干净——信息不重复，维护点唯一。
