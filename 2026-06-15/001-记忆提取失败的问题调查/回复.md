看下来了，两种情况，都不严重。

## 两次失败追溯

**第一次 (13:41:36) — LLM 返回空**
```
parse_llm returned empty, treat as parse_fail
```
LLM 调用了但返回了空内容（可能是 proxy 的 deepseek-v4-pro 模型偶发返回空结果），不是代码错误，已 requeue。

**第二次 (13:42:08) — 重试耗尽**
- 队列被重试了 3 次（`retry_count=3 >= 3, discard`），最终丢弃
- 但同时 item[1]（当前 session `d1617867-8de` 的内容）**已经 `sub status=ACCEPT` 成功存储**

## 关键 bug：`consecutive_parse_fails` 计数器是错的

在 `consume_queue()` 的这段代码（`session_knowledge.py:644-645`）：

```python
if fail_count > 0:
    prev_status["consecutive_parse_fails"] = prev_status.get("consecutive_parse_fails", 0) + fail_count
```

问题是：它**跨执行累加**，且同一个 run 内即使有成功项目也不会重置。第二次 consume 时：
1. item[0] 已重试 3 次 → 丢弃 → fail_count=1
2. item[1] **ACCEPT 成功** → fail_count 仍然=1
3. 读取上次状态 `consecutive_parse_fails=1` → `1 + 1 = 2`

但实际上最后一次（item[1]）是成功的，没有本质问题。这个计数在下一个 SessionStart 还是被正确清零了（`session_start.py:78`）。

## 结论

你看到的"2次解析失败"警告有点"虚高"——实际只有一条 4 天前的旧项目被丢弃（已重试 3 次），当前 session 的记忆已经被成功提取了。计数器跨 consume 累加的逻辑有缺陷，但功能上不影响。要修吗？
