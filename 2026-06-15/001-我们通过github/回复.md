这是个好问题。先理清现状，再比较方案。

## 当前已有的互锁机制

- **GitHub PR / Docker 镜像**：跨项目 API 变更靠 CI 和版本号卡住，是"事后验证"层
- **memory-mcp service-catalog**：记录了项目间依赖关系，是"项目级"粒度
- **跨项目变更规则（CLAUDE.md）**：三步走（查下游 → 改代码 → 同步消费者），是"流程"层

缺失的一环是**项目内文件级**的依赖认知——改 A 文件时知道 B、C、D 会受影响。

## 三种方案比较

### 方案 A：记忆存储文件架构

每个项目存一份 `architecture` 段到 memory-mcp，描述目录职责和关键文件依赖。

**不可行。** 原因：
- claudetalk 约 40+ 源文件，model-proxy 约 25+，ops-daemon 约 15+，全量记忆轻松破万 token
- 每次改文件就要更新记忆，维护成本 > 收益
- 记忆适合存"为什么这样设计"（pitfalls、设计决策），不适合存"有什么文件"（代码本身就是真相）

### 方案 B：静态依赖分析工具

用 `madge`（TS）或 `pydeps`（Python）生成依赖图，按需查询。

```
madge --image dep.svg src/index.ts   # 可视化
madge --json src/index.ts             # 机器可读
```

**优点**：零维护，永远准确，覆盖 import/require 级别的依赖。
**局限**：只管显式导入，不管隐式依赖（配置文件字段名、MCP tool schema、消息格式）。

### 方案 C：功能切面映射（推荐）

不存文件列表，存的是**功能切面 → 涉及文件模式**的映射。粒度比"项目"细，比"文件"粗。

以 claudetalk 为例：

```yaml
# 存在 memory-mcp claudetalk/architecture 中
functional_surfaces:
  message_ingress:
    desc: "消息入口：接收飞书消息 → 分类 → 路由到 Claude"
    entry: src/channels/feishu/index_feishu.ts
    touchpoints:
      - src/core/classifier.ts        # 消息分类
      - src/core/claude.ts            # Claude API 调用
      - src/core/session.ts           # 对话 session 管理
    config: .env (FEISHU_APP_ID, etc.)
    
  card_action_handling:
    desc: "飞书卡片按钮回调处理"
    entry: src/channels/feishu/index_feishu.ts (WSClient card.action.trigger)
    touchpoints:
      - src/core/classifier.ts        # fallback 分类
      - src/mcp-server.ts             # methodology card → MCP tool 调用

  mcp_tools:
    desc: "MCP Server 暴露的工具接口"
    entry: src/mcp-server.ts
    touchpoints:
      - src/tools/*.ts
    consumers: [claude-code]          # 外部消费者
```

一条记忆大概 500-800 字符，5-8 个功能切面覆盖一个中型项目，总计 3-4K 字符——不会太大。

## 建议的落地方式

1. **每个项目加一个 `architecture` 记忆段**，只存功能切面映射（上面的格式），不存文件列表
2. **改代码前查两样东西**：`memory_recall` 查功能切面 + `madge` 或 grep 查实际 import 链
3. **功能切面变更时才更新记忆**（新增/删除切面、调整 touchpoints），比每次改文件更新成本低一个数量级
4. **不存到 CLAUDE.md**——CLAUDE.md 是给人读的入口文档，记忆是给 AI 检索的，用途不同
