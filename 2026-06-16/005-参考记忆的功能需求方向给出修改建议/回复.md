好，我看完了整条链路。系统里有三个相关模块：

- **`session_knowledge.py`**（SessionEnd hook）— 提取入口
- **`_storage.py`**（memory-mcp 核心）— 存取 + health_sweep + LLM audit
- **`daily_audit.py`**（ops-daemon 每日审核）— 对 pending 记忆做 4 维度审查

以下是我基于功能需求的分析。

---

## 问题诊断

### 1. Extraction prompt 过载 — 一件事让 LLM 干太多

`session_knowledge.py` 单次 LLM 调用同时做：
- 判定 ACCEPT/REJECT/UPDATE
- 定 namespace/segment
- 写 content（含 Why + How to apply）
- 写 intended_recall_query
- memory_recall 查已有记忆 → 注入 prompt → 要求对比合并
- same-session 扫描已有记录 → 再注入

结果：prompt 长达 60 行，deepseek-v4-pro 昂贵且输出质量不稳定。

### 2. 审查反馈环断裂

- `_storage.py._ensure_audit_record()` 写 `review_status: pending`
- `daily_audit.py` 读 pending 做审核 → 读的人很少，结果只发一个飞书卡片
- 提取质量好坏没人看，没人修

### 3. 记忆提取粒度太细

每次 SessionEnd 提取 1-5 条独立记忆。多为低价值调试记录，高价值洞察被稀释。每条都走 jaccard 相似度去重 → 拖慢写入。

### 4. audit-record 字段膨胀

`source_session / extraction_date / intended_recall_query / review_status / review_date / review_verdict / review_note / merge_target` — 8 个字段，实际只有 `review_status` 被 daily_audit 用到一次，其他闲置。

---

## 建议修改方向

### Phase 1 — 简化提取 hook

SessionEnd hook **退出 curation 职责**，只做两件事：

1. 判定会话是否有提取价值（bugfix / 架构决策 / 新依赖 / 踩坑根因）
2. 有则写一条结构化记录到 staging 队列（不直接 memory_store）

取消：
- memory_recall 查重（延迟到 curation 阶段）
- same-session 扫描
- audit-record 注入
- _storage import

### Phase 2 — 异步精选管道

新增一个 `curator` 批处理任务（ops-daemon 每日或每隔几小时触发）：

1. 读 staging 队列
2. 调 memory_recall 获取已有记忆
3. LLM 决策：合并到已有 / 新建 / 丢弃（单任务，用 flash，便宜）
4. 写 _storage.memory_store()
5. 生成质量报告

### Phase 3 — 去掉废弃物

- 从 `_storage.py` 移除 audit-record 相关逻辑（_ensure_audit_record / _extract_audit / AUDIT_RECORD_RE）
- `daily_audit.py` 功能并入 curator 管道后废弃
- memory_recall 返回格式简化（现在的 format 直接暴露给下游了）

---

## 收益

| 维度 | 现在 | 改后 |
|------|------|------|
| prompt 复杂度 | 60+ 行全量决策 | 2 个独立子任务，各 10 行 |
| 模型成本 | depthseek-v4-pro | pro → (提取) + flash → (精选) |
| 提取质量 | 无反馈 | curator 可做聚合去重 |
| 接口简化 | 一堆跳过逻辑 | 清晰的 staging → curated 两步 |
