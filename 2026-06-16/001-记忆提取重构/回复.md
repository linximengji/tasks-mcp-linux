# 记忆提取重构方案

## Phase 1 — 简化提取 hook
SessionEnd hook 退出 curation 职责，只做两件事：
1. 判定会话是否有提取价值（bugfix/架构决策/新依赖/踩坑根因）
2. 有则写一条结构化记录到 staging 队列（不直接 memory_store）
不调 LLM、不 import _storage、不发飞书、不写 audit

## Phase 2 — 异步精选（Spawn 驱动）
复用 SessionEnd 的 subprocess.Popen(DETACHED_PROCESS) 模式：
- SessionEnd 触发时 spawn curator 子进程
- 读 staging 目录所有 .json 文件，批处理
- LLM (flash, 便宜) 决策：合并到已有/新建/丢弃
- 写 _storage.memory_store()/memory_update()
- 成功 → 移入 staging/processed/，失败 → 不动，下次 SessionEnd 重试
- 无 cron、无阈值、无人工延迟

## Phase 3 — 清理废弃物（保留兼容）
- `_storage.py`：memory_store() 中去掉 _ensure_audit_record() 这一行调用 → 新记忆不再写 audit
- 保留 _strip_audit()/_extract_audit()/AUDIT_RECORD_RE（jaccard 去重 + memory_update 兼容需要）
- 保留 memory_review MCP tool（ops-daemon health check 引用）
- 废弃 daily_audit 定时调度，保留代码文件
- memory_recall 返回格式不动（MCP tool 被 LLM agent 调用）
