### Phase 1 — 简化提取 hook

SessionEnd hook **退出 curation 职责**，只做两件事：

1. 判定会话是否有提取价值（bugfix / 架构决策 / 新依赖 / 踩坑根因）
2. 有则写一条结构化记录到 staging 队列（不直接 memory_store）

取消：
- memory_recall 查重（延迟到 curation 阶段）
- same-session 扫描
- audit-record 注入
- _storage import

意见：这一步没有什么问题

### Phase 2 — 异步精选管道

新增一个 `curator` 批处理任务（ops-daemon 每日或每隔几小时触发）：

1. 读 staging 队列
2. 调 memory_recall 获取已有记忆
3. LLM 决策：合并到已有 / 新建 / 丢弃（单任务，用 flash，便宜）
4. 写 _storage.memory_store()

点评和问题：这个执行时机再给一个更好的策略

### Phase 3 — 去掉废弃物

- 从 `_storage.py` 移除 audit-record 相关逻辑（_ensure_audit_record / _extract_audit / AUDIT_RECORD_RE）
- `daily_audit.py` 功能并入 curator 管道后废弃
- memory_recall 返回格式简化（现在的 format 直接暴露给下游了）

意见：memory recall这个是给LLM调取记忆用的，简化规则不能影响这个核心用途
audit record 有没有服务依赖查清楚。

整体更新一版方案
