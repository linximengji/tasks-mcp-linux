你说得对，这里有设计问题。看代码是怎么写的：

**`main.py:532-534`** — 每次 daemon boot 无条件调用 `spawn_claudetalk()`：

```python
if cc.get("claudetalk", {}).get("enabled", False):
    spawn_claudetalk()   # ← 没有 TCP 检查，没有判断"是否需要"
```

对比同一段代码对 proxy 的处理（`main.py:521-527`）：

```python
if cc["proxy"]["enabled"]:
    if not _tcp_check("127.0.0.1", 4000):   # ← 先确认真挂了才起
        spawn_proxy(4000)
```

**问题不是 `_is_pid_alive` 不准**（那个 guard 多数情况下能拦），而是架构上的方向反了：

1. **依赖方向倒置** — daemon 是观察者，不是管理器。子进程（proxy、claudetalk、MCP server）独立于 daemon 运行，daemon crash 不应该影响它们。但 boot 代码主动去"起"它们，把子进程生命周期绑到了 daemon 上。

2. **两个平行路径冲突** — boot 时 spawn + 60s 周期 check-repair，两条路径做同一件事。当 daemon 频繁 crash 时（6/22 那晚），boot spawn 反复触发，但明明等 60s 后的 check 周期处理就够了。

3. **没有 TCP 验证** — proxy 有 `_tcp_check`，MCP server 也有 `_tcp_check(9877)`，但 claudetalk 直接走 `spawn_claudetalk()` 内部 PID 文件 + tasklist 检查。PID 文件在僵尸清理时可能已经被标记为死进程，导致不必要的重启。

**正确的设计应该是**：daemon boot 只做自己 — 初始化状态、注册 check、进循环。子进程死活由周期 check 检测 + repair 子系统处理。如果 claudetalk 真挂了，60s 内 check 会报 `status: "stopped"`，repair 机制会拉起来。boot 时不需要 extra 逻辑。

要改吗？删掉 `main.py` 里 `spawn_claudetalk()` 和 `spawn_mcp_server()` 那几行，proxy 的 boot spawn 因为有 TCP check 相对安全但也可以删掉保持一致，都交给 check-repair 循环去管。
