## 根因分析

### 直接原因：claudetalk 优雅退出（exit code 0）

从所有可获取的信息看：
- **无 crash marker** → exit code = 0（`_watch_exit` 只在非零退出才写 marker）
- **无错误日志** → 没有触达 `unhandledRejection` / `uncaughtException` handler
- **无定时器自动退出** → 没有 idle-timeout 或 watchdog 机制
- **无外部 kill** → zombie reaper 明确排除 claudetalk
- **无超时相关退出** → timeout 只 kill 子进程（claude-stream），不杀主进程

唯一能 cause exit code 0 的代码路径是 `drainThenExit` → `process.exit(0)`，触发条件只有 **SIGTERM / SIGINT / `/restart` 命令**。

### 最可能的情境

Claudetalk 被某种原因 reload/restart 了（可能是 daemon 之前的 `on_checks` 触发的 `spawn_claudetalk(kill_first=True)`，或是 `--restart` 命令），旧进程 `drainThenExit` 优雅退出，新进程可能没有成功启动。但具体谁发的信号由于时间线交叠无法确定。

### 根本原因（修复链断裂）

不管死因是什么，**daemon 在两次修复失败后永久放弃**才是让你感觉到"动不动就死"的原因：

1. 10:35:28 — claudetalk 首次被发现 stopped
2. 10:36:04 — 第一次修复失败（30s 等待超时）
3. 10:38:46 — 第二次修复失败，`repair_max=2` 耗尽
4. `repair_exhaust_notified` 永久锁定 claudetalk
5. 此后每 60s 空转记录 `claudetalk_stopped`，**再也没有重试过一次**

```python
# main.py:287 — 只在正常状态才清除耗尽标记
if not is_fail:
    repair_exhaust_notified.discard(name)

# main.py:300-311 — 耗尽后永久跳过
if attempts >= repair_max:
    repair_exhaust_notified.add(name)  # 只加不减
    continue  # 永久跳过
```

### 修复要点

两个层面：

**1. repair 耗尽自动解锁（关键）** — 加 `repair_cooldown_minutes` 配置，耗尽 N 分钟后自动移出 `repair_exhaust_notified`，给下一次修复机会。

**2. 提高 recovery 成功率** — `repair_max` 从 2 提高到 3-4，或加少量重试间隔（当前 repair 连续密集重试，30s 超时太紧——claudetalk 从启动到可用可能超过 30s）。
