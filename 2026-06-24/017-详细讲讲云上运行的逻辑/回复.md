## 逐层拆解

### 第一层：入口（用户怎么发起对话）

三个入口，效果等价——都触发 CC 进程：

| 入口 | 实现 | 适用场景 |
|------|------|---------|
| 飞书消息 | ClaudeTalk 收 Webhook → spawn `cc -p "消息"` | 手机随时用 |
| Web UI | remote-agent 已实现，浏览器打开 | 电脑不在身边 |
| SSH | `ssh cloud-server "cc"` 进 tmux 会话 | 深度操作、改配置 |

### 第二层：CC 进程生命周期

```
飞书消息到 ──► ClaudeTalk 收到 webhook
                  │
                  ├─ cc --resume -p "用户的飞书消息"
                  │     │
                  │     ├─ CC 启动，加载 settings.json
                  │     ├─ 连接 MCP servers（memory-mcp:3003, proxy:4000, tunnel→blender/p1s）
                  │     ├─ 加载上次 session 上下文 (--resume)
                  │     ├─ 推理 → 工具调用 → 返回结果
                  │     ├─ CC stdout → ClaudeTalk → 飞书回复
                  │     └─ 进程退出
                  │
                  │ 5分钟后用户又发消息 ──►
                  │
                  └─ cc --resume -p "继续刚才的"
                        │
                        └─ 有 --resume，上下文接上，像同一个对话
```

关键点：**CC 不是守护进程，是短生命周期进程。** 对话上下文靠 `--resume` 衔接，对话历史存在 `~/.claude/sessions/` 里。

### 第三层：MCP 服务——分为两类

**云上本地启动的（CC 进程启动时一并拉起）：**

| 服务 | 方式 | 说明 |
|------|------|------|
| model-proxy | `command: python` 本地子进程 | CC 直接启动，API 密钥只在云上 |
| memory-mcp | `command: node` 本地子进程 | 持久记忆，和 CC 同机 |

**settings.json 片段：**
```json
{
  "mcpServers": {
    "memory-mcp": {
      "command": "node",
      "args": ["/opt/projects/memory-mcp/dist/index.js"]
    },
    "model-proxy": {
      "command": "python",
      "args": ["/opt/projects/proxy/model_proxy.py"]
    }
  }
}
```

**远程连接本地硬件的：**

| 服务 | 方式 | 路径 |
|------|------|------|
| blender-mcp | HTTP/SSE over Cloudflare Tunnel | `https://blender.linximengji.com/sse` |
| p1s-mcp | HTTP/SSE over Cloudflare Tunnel | `https://p1s.linximengji.com/sse` |

**settings.json 片段：**
```json
{
  "mcpServers": {
    "blender-mcp": {
      "url": "https://blender.linximengji.com/sse"
    },
    "p1s-mcp": {
      "url": "https://p1s.linximengji.com/sse"
    }
  }
}
```

这些 URL 走 Cloudflare Tunnel → 穿透到本地 Windows 上对应的 localhost 端口。MCP 协议本身是 HTTP + SSE，Tunnel 天然支持。

### 第四层：本地 Windows 的角色

本地 Windows 不再是主控端，降级为**硬件资源池**：

```
本地只做两件事：
  1. 运行硬件绑定的 MCP 服务（Blender/P1S/Playwright GUI）
  2. 通过 cloudflared 把它们暴露到公网（已有 Cloudflare Access 保护）

不再跑：
  - CC CLI          → 迁云
  - model-proxy     → 迁云
  - memory-mcp      → 迁云
  - ops-daemon      → 迁云（进程守护）
  - ClaudeTalk      → 迁云（入口逻辑）
  - 项目代码        → git clone 到云
```

### 第五层：ops-daemon 在云上的职责

云上的 ops-daemon 简化为只管理云上服务：

```
ops-daemon (云版)
  ├─ process_manager: model-proxy (4000), memory-mcp (3003)
  ├─ health_checks: 检查 proxy health, memory-mcp status
  ├─ proxy_switch: 主备切换（如果 proxy 挂了）
  └─ 不再管: blender, p1s, cloudflared 隧道（隧道本身在本地）
```

## 数据流完整示例：用户飞书消息 "用 Blender 做个立方体"

```
1. 飞书 Webhook → 云上 ClaudeTalk
      │
2.    ├─ ClaudeTalk spawn: cc --resume -p "用 Blender 做个立方体"
      │
3.    ├─ CC 进程启动
      │   ├─ 加载 settings.json → 注册 4 个 MCP server
      │   ├─ 启动本地子进程: model-proxy (4000), memory-mcp (3003)
      │   ├─ 连接 HTTP SSE: blender*.linximengji.com, p1s*.linximengji.com
      │   └─ --resume 加载上次对话上下文
      │
4.    ├─ 推理阶段
      │   ├─ API 请求 → model-proxy:4000 → DeepSeek API
      │   ├─ 决定调用 blender_execute_code
      │   └─ model-proxy 处理 token 计费、错误重试
      │
5.    ├─ 工具调用: blender_execute_code(code="import bpy; bpy.ops.mesh...")
      │   ├─ HTTP POST → https://blender.linximengji.com/sse
      │   ├─ Cloudflare Tunnel → 本地 Windows blender-mcp
      │   ├─ Blender 执行 → 创建立方体
      │   └─ 返回结果 → Tunnel → 回云上 CC
      │
6.    ├─ 继续推理 → 可选 blender_get_screenshot 验证
      │   └─ 同上 Tunnel 路径
      │
7.    ├─ CC stdout: "已在 Blender 中创建立方体"
      │
8.    └─ ClaudeTalk → 飞书回复用户

整个过程 3-8 秒，CC 进程退出。下一次消息再 spawn 新进程。
```

## 为什么这是完整的系统

云上有：
- **推理引擎**：CC CLI
- **模型路由**：model-proxy
- **持久记忆**：memory-mcp
- **入口网关**：ClaudeTalk（飞书/Web/SSH）
- **进程守护**：ops-daemon
- **代码仓库**：所有项目 git clone

它唯一不能做的事是需要专用硬件的事（GPU 渲染、局域网打印）。这些由本地 Windows 作为外设提供。

类比：**云服务器是大脑 + 记忆 + 嘴巴，本地 PC 是手和眼睛。** 你从飞书说话 → 大脑处理 → 需要用手时通过神经（Tunnel）指挥手 → 手执行 → 结果回大脑 → 嘴巴回复你。
