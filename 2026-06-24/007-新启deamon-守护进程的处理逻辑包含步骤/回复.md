你的理解基本正确，但有一处时序差异——zombie scan + repair + PID 列表更新不是 boot 阶段的步骤，而是 **进入 daemon.run() 主循环后的第一个 60s 周期内** 发生的。

完整的启动流程按阶段拆：

**Boot 阶段（同步，~100ms）**

| 步骤 | 代码 | 说明 |
|---|---|---|
| 1. PID 原子锁 | `main.py:161-189` | `msvcrt.locking` 防止重复实例，检测旧 PID 是否存活 |
| 2. 初始化状态 | `191-195` | StateStore / BaselineEngine / AlertManager |
| 3. 创建 BaseDaemon + 写 heartbeat | `197` | `heartbeat_path` 文件写入时间戳 |
| 4. 注册所有 check | `199-239` | proxy / services / system / ssl / logs / **zombies** / claudetalk / mcp_server / cloudflared 等 |
| 5. 注册 on_check_complete 回调 | `253-412` | proxy_auto_switch / **on_checks(repair引擎)** / **claudetalk_lifecycle** / **repair_proxy_backup** / **_write_pids** |
| 6. 启动 scheduler（定时任务） | `509` | daily_report / semi_report |
| 7. ~~boot spawn~~ | `512-515` | — **已删除，子进程全由周期 check 管** |
| 8. 启动 restart_watcher | `517` | 轮询 `.restart-*` 标记文件（safekill 触发） |
| 9. 进入 daemon.run() | `520` | 开始 60s 主循环 |

**第一个 check 周期（~60s 内）**

| 事件 | 说明 |
|---|---|
| **zombie scan** | `check_zombies()` — 扫描 psutil 中 p1s-mcp/blender-mcp 等 MCP 进程，超过 120min 未更新的 kill |
| **各服务健康检查** | proxy(TCP+HTTP)、claudetalk(PID+crash marker)、mcp_server(TCP+HTTP)、services、system 等 |
| **repair on_checks** | `REPAIR_REGISTRY` 查找 status 为 down/stopped 的服务 → 自动拉起 |
| **_write_pids** | 将当前各服务 PID 写入 `protected_pids.json`（protect_services hook 读此文件） |
| **proxy_auto_switch** | proxy manager 评估是否需要主备切换 |

所以新 daemon 启动后对子进程的完整处理是：**不主动拉，但 60s 内一定能检到异常并修复**，同时对僵尸进程做周期清理。你说的"僵尸审查→僵尸处理→repair→更新守护列表"全部对，只是它们发生在第一个周期而不是 boot 瞬间。
