全部改动已就位，已编译通过。总结这个回合加的和已有的所有 instrumentation：

---

**已生成的 marker 文件清单** (都在 `.claudetalk/`):

| 文件 | 触发时机 | 内容 |
|------|----------|------|
| `startup-marker.json` | 启动时每步写入，前120s每10s心跳 | phases 数组 + ts |
| `exit-marker.json` | 每次退出覆写 | type(exit/beforeExit/process.exit/...)、uptime、startup_phases |
| `crash.marker` | 每次退出同步写 | plain timestamp（daemon check 兼容格式） |
| `claudetalk-{date}.log` | 每次退出追加 `[EXIT] {JSON}` | 日志文件中可查 |
| `claudetalk.log` | 最早阶段写 `[BOOT]` | initLogFile 前的最早信号 |

**退出路径全覆盖**:

| 退出原因 | 捕获方式 | exit-marker.json type |
|----------|----------|----------------------|
| SIGTERM → drainThenExit → process.exit(0) | process.exit 拦截 | `process.exit` + stack trace |
| main().catch → process.exit(1) | 同上 | `process.exit` + stack trace |
| unhandledRejection | handler | `unhandledRejection` + detail |
| uncaughtException | handler | `uncaughtException` + stack |
| Event loop 无活可干 | beforeExit | `beforeExit` + startup_phases |
| 兜底（任何 exit 路径） | process.on('exit') | `exit` |

重启 daemon 后，下次 claudetalk 死后直接看 `.claudetalk/exit-marker.json` 即可知道 root cause。如果看不到这个文件，说明进程被 OS 强行杀死（`taskkill /F` 或其他），连 Node.js 的 exit 回调都没机会跑。
