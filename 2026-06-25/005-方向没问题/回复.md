## 完整影响分析：把 WebSocket 拆进 MCP Server

### 当前链路

```

             ├─ 接收消息 → onMessage handler (index.ts)
```

### 最难处理的三个问题

**1. card.action 3 秒窗口**

目前 FeishuClient 在 `start()` 时给 fly SDK `EventDispatcher` 注册了 `card.action.trigger` 处理器。卡片按钮需要**在 3 秒内**返回 `{ toast: {...} }` 给飞书 API。这个 handler 里包含了：

- `mark-task-done` → 写本地的 `index.json`（纯文件 I/O，很快，3s 完全够）
- `confirm-archive` → 调用 `_lastConvGetter` 拿最近对话、调档案功能（需要 claudetalk 内数据）
- `approval-action` → 读审批缓存、返回 toast
- `execute` → spawn 子进程调 Claude CLI（异步，3 秒内只返回 "正在执行"）

**卡住的地方**：`mark-task-done` 可以在 MCP server 里安全同步执行（纯文件 I/O）；但 `confirm-archive` 需要访问 claudetalk 内存中的 `_lastConvPair`。

→ 方案：MCP server 同步执行 `mark-task-done` 和 `toast` / `dismiss` 等简单 action；`confirm-archive` 和 `execute` 返回 "将移交给bot处理" toast，消息入队列等 claudetalk 消费。

**2. Peer-message 轮询**

`startPeerMessagePolling()` 每 5s 扫 `bot_{profileName}.json` 文件。多个 bot 的 FeishuClient 之间通过这个机制通信。如果 FeishuClient 在 claudetalk 里而 WebSocket 在 MCP server 里：

- peer-message **文件写入**发生在 `sendAndWritePeerMessage()`（FeishuClient 的发送方法），这条链路已通过 MCP server 发送，但写入由 claudetalk 完成
- peer-message **读取轮询**在原来的 FeishuClient 里

问题：claudetalk 崩了 → peer-message 轮询停 → 其他 bot 发来的消息等 claudetalk 恢复后才能处理。

→ 方案：peer-message 轮询搬到 MCP server，读取后入 `/pending-events` 队列。

**3. claudetalk 重启后 dequeue**

MCP server 队列中的消息在 claudetalk 恢复后如何消费？

- 简单轮询：claudetalk 拉 `/pending-events`，有消息就处理
- 问题：claudetalk 恢复后怎么知道"之前正在处理什么"？会话上下文呢？
- 实际上 claudetalk 本身无状态，session 是 Claude CLI 维护的，不存在"恢复进行中会话"的问题。claudetalk 恢复后，MCP server 队列里的消息按顺序发给 claudetalk，每条都是新请求。

### 关键风险点

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| 拆分后 MCP server 崩 = 两路全断 | 高 | MCP server 是纯 I/O 几乎不崩；加 ops-daemon crash 检测 |
| card.action 跨进程慢响应 | 中 | 分两类处理：本地快的直接回，需要 claudetalk 的异步队列 |
| Peer-message 通信错乱 | 中 | 统一归 MCP server 管理 |
| 飞书 token 竞争（双进程各自缓存）| 低 | 各自独立缓存，token 是只读凭证，无竞争问题 |
| chat-members.json 同时写入冲突 | 低 | JSON write 一般是原子性的，文件不大 |
| /pending-events 内存无限增长 | 中 | 加上限 + TTL 淘汰 |
| claudetalk 恢复后重复拉旧消息 | 中 | 加 cursor/ack 机制 |
