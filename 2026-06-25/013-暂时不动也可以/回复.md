## C盘增长源头分析

先看各个项目的**性质**— 静态 vs. 动态增长：

### 按"增长速度"排序

#### 1. Docker 废弃卷 — 4.7 GB（100% 可回收！）

```
docker system df 显示:
Local Volumes: 7 个卷 / 4.729 GB 总量 / 4.729 GB (100% RECLAIMABLE)
```

**7个卷全是孤立的**——没有容器在用！Docker pull 新镜像时留下的老镜像层也占着空间。容器跑得越久，pull 更新越频繁，空间涨得越快。

#### 2. 剪映（JianyingPro）— 6.6 GB，其中 4.8 GB 是用户数据

```
Apps:       1.8 GB  ← 安装文件，一次写入
User Data:  4.8 GB  ← 素材缓存、草稿、渲染中间结果，**持续增长**
```

每做一个视频，素材缓存和渲染文件就往 C 盘 User Data 里写。如果你用剪映，这个是 C 盘最大的 "日常消耗者"。

#### 3. Docker 镜像增量

| 镜像 | 大小 | 说明 |
|------|------|------|
| presenton | 2.6 GB | 镜像体积大，更新时下载新层 |
| pact-broker | 852 MB | 版本更新频繁 |
| searxng | 259 MB | 相对稳定 |
| jaeger | 85 MB | 小 |

每次 docker compose pull 更新，都会在 C 盘追加新层。**Docker 整个 vhdx 6.6 GB 就是镜像 + 废弃卷 + 镜像层的总和。**

#### 4. 其他（中低增长）

| 项目 | 大小 | 增长原因 |
|------|------|---------|
| Temp 临时文件 | ~755 MB | VS Code、Claude Code 运行时临时文件 |
| VS Code 扩展 | 387 MB | 装新扩展或更新时增长 |
| Downloads | ~1.9 GB | 下载的安装包（很多已无用）|

### 结论：两大核心增长源

```
1. Docker 废弃卷          4.7 GB  →  可立即回收（100% orphaned）
2. 剪映用户数据            4.8 GB  →  持续增长，建议迁 D 盘
```
**这两个就占了 9.5 GB。** 其他（Temp、VS Code、Downloads）增长慢，影响小。

### 建议立即操作

不需要动 Docker 数据迁移，只要清理就可以了：

```
docker volume prune -f     # 回收 4.7 GB 废弃卷
docker image prune -a      # 回收未用镜像
```

加上上午已经完成的 npm/pip/uv 缓存迁移（8.5 GB），两下里加起来就是 **~13 GB**，C 盘 74% → ~68%。你来决定要不要清 Docker 的废弃卷？
