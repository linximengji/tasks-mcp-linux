## 方案审核评价

### 1. 问题诊断偏差

方案说 claudetalk 崩了导致 WebSocket 断连，"彻底分离"是解决方案。但这个推理有跳跃：

- 你当前的**实际痛点**是 **3600s 退避**——claudetalk 15 秒内崩 5 次就停 1 小时
- 方案却选择解决一个"claudetalk 重启时消息不丢"的问题

**claudetalk 每次崩都是 bug，不是常规行为。** WebSocket 断开->飞书 SDK 重连->恢复，总窗口不超过 30-60s。这 30s 内用户看到的是"bot 红点已读但不回"，不是好消息但远没有 1h 严重。

### 2. "MCP server 几乎不崩"是危险的假设

方案写 "MCP server 是纯 I/O 几乎不崩"，但 `mcp-server.ts` 已经是 **650 行**的 HTTP server + SSE 协议 + multipart 文件上传。把 FeishuClient **1684 行**的 WebSocket + 事件去重 + peer-message 轮询 + 文件下载逻辑塞进去，变成了 **2300+ 行的单体**——一旦引入 bug，**上下行全崩**。当前 MCP 崩了只影响下行通知（CC→飞书），用户不感知。

### 3. 被低估的复杂度

方案估了 11 个文件，实际远不止：

| 实测发现 | 影响 |
|----------|------|
| `channel.sendMessage()` 在 index.ts 中 **25+ 调用点** | 每个都在等 `channel.sendMessage` → 现在是直接调飞书 API，改后要走 HTTP 到 MCP server |
| `(channel as any).sendMarkdownCard()` 3 处 | 新增的 RPC 接口 |
| `channel.sendMessageWithId()` + `editMessage()` 行 966-991 | 对话流式输出需要拿到 message_id 后 continue editing，跨进程时窗口竞争很微妙 |
| `approval-handler.ts` 第 23 行的 `pendingApprovals` Map | **进程内内存状态**。card.action 回调和审批结果确认都在这里，跨进程后要么搬要么重设计 |
| `peer-message.ts` 的原子写入协议 | 当前**写入方是 MCP (通过 feishu_notify 工具) ，轮询方是 claudetalk**。搬了之后依赖链更复杂 |
| `index_feishu.test.ts` 168 行测试 | 需要完全重写，测试对象从 FeishuClient 变成 HTTP 客户端 |

**保守估计 18-20 个文件，不是 11 个。**

### 4. card.action 3 秒窗口的处理不合理

方案说"简单的 MCP 直接回，复杂的丢队列"。但看看 `handleCardAction` 的实际内容：

- `mark-task-done`：写 `index.json`，确实快
- `confirm-archive`：需要 `_lastConvPair` ← 这是 **index.ts 的内存变量**，MCP server 根本访问不到
- `approval-action`：读写 `pendingApprovals` Map ← 这是 **approval-handler.ts 的模块级变量**

"消息入队列等 claudetalk 消费" → 消费时 3 秒窗口已过，飞书会显示"请求超时"。用户的卡片会停留在灰色 loading 状态直到手动刷新。

### 5. 真正的替代方案

对比两个方向的成本：

| | 方案 A：改退避策略 | 方案 B：WebSocket 拆入 MCP |
|---|---|---|
| 文件变动 | **1 个** (`repair.py` 改几行) | **18-20 个** |
| 新增风险 | 无（降级退避时长短了，只会更好） | 高（2300+ 行单体 MCP、进程间内存状态、流式编辑窗口竞争） |
| 效果 | 退避缩短，尽快恢复 WebSocket | claudetalk 重启期间消息攒在队列不丢 |
| 测试验证 | 改 3 行 + 跑一次 crash 模拟 | 需要完整集成测试 |

方案 A 只需要把 `repair.py` 的退避从 `3600s → 300s 阶梯式`，同时在退避触发时 `notify.py` 发一条飞书通知告诉用户"bot 崩了正在恢复"。

### 结论

这个方案方向理论上 OK，但**当前场景下是 over-engineering**。真正的问题只占了 10%，方案的 90% 都在解决一个不存在的"消息不丢"需求——claudetalk 正常运行时不丢消息，crash 恢复后 Claude CLI session 的上下文才是真正的问题（而不是中间哪条消息没处理）。

推荐走方案 A，先解决真正的痛点。如果以后出现"单 bot 实例不够用、要横向扩展"的场景，那时候再拆 WebSocket 也不迟，而且有更清晰的驱动因素。
