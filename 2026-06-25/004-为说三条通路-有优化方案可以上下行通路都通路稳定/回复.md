你说得对，我加的第三条通路 C（notify.py）其实不该算一条独立的"通路"——它只是 daemon 内部的告警出口，跟"手机和 CC 对话"这件事无关。回到你的原始问题：**两条真正面对用户的飞书通路**。

## 重新聚焦：两条通路

```
通路 A (下行 CC→手机): feishu-notify MCP (port 9877)
通路 B (上下行 手机↔CC): claudetalk FeishuClient (WebSocket)
```

**上行**（手机发消息给 bot）只走通路 B 的 WebSocket。这就是脆点：claudetalk 崩了，上行全断。下行可以临时走通路 A，但回复语境丢失，基本没用。

## 核心问题

通路 B 的 WebSocket 长连接绑定在 claudetalk 进程内。进程需要重启 → WebSocket 断开 → 飞书 SDK 自动重连至少 30s → 这期间消息全丢。

## 优化方案：把 WebSocket 拆进 MCP Server

```
现在:
  MCP server (9877)      = 只发不下收，空占一个端口
  claudetalk FeishuClient = WebSocket + 命令处理混合
  
改为:
  MCP server (9877)      = WebSocket 长连接 + 发送 API → 飞书唯一网关
  claudetalk (无端口)     = 纯命令处理器，从 MCP server HTTP 拿消息
```

改动量：

| 组件 | 变更 |
|------|------|
| `mcp-server.ts` | 加 FeishuClient 逻辑（从 index_feishu.ts 搬），WebSocket 接收→HTTP 转发给 claudetalk |
| `index.ts` | 删 FeishuClient 导入，改为轮询/长轮询 MCP server 拿消息 |
| `index_feishu.ts` | `FeishuClient` 类整体移到 mcp-server.ts，不再被 cli.js 实例化 |
| `process_manager.py` | 启动顺序不变，但 `spawn_claudetalk` 不再需要关心飞书凭据 |

效果：

- claudetalk 崩溃/重启 → WebSocket 不中断，MCP server 把消息暂存队列
- claudetalk 恢复后从队列消费，用户无感知
- MCP server 崩溃 → 上下行全断，但它是纯 I/O 代理几乎不崩
- 飞书凭据集中在 MCP server 一处，claudetalk 不再需要

这就是把 MCP server 做成**真正的飞书网关**，claudetalk 退化为无状态命令处理器。你觉得方向对吗？
