## 架构分离情况分析

你的系统里现在有**两条完全独立的飞书通讯通道**：

### 通道 A：Feishu-Notify MCP（与 claudetalk 无关）

```
CC Session Hook → feishu-notify MCP 服务 → 飞书开放平台 API
```

当前对话中可调用的 `feishu_send_card`、`feishu_send_markdown`、`feishu_send_text`、`feishu_status` 就是这组工具。它由独立的 MCP server 提供（配置在 ~/.claude 的 MCP 注册表里），由 Claude Code 进程生命周期管理，不依赖 claudetalk。

**claudetalk 崩溃/重启对通道 A 的通讯无任何影响。** 手机飞书仍然能收到这条路过来的通知（如 SessionEnd 知识提取通知）。

### 通道 B：Claudetalk FeishuClient（依赖 claudetalk）

```
手机飞书 → 飞书开放平台 → WebSocket → claudetalk FeishuClient → callClaude()
```

claudetalk 进程内的 `FeishuClient` 维护着飞书 WebSocket 长连接，用官方的 `@larksuiteoapi/node-sdk` 建立。这是"手机发消息给 bot → claudetalk 处理 → 回复"这条路径。

**claudetalk 崩溃直接影响这条路径**——WebSocket 断开，手机发的消息收不到回复。但 ops-daemon 有 crash 检测和自动重启，15 秒内 5 次快速崩溃会触发退避（停 3600 秒不重试）。

### 通道关系总图

```
手机飞书用户
    ├── 发消息给 bot ──→ 通道 B (claudetalk FeishuClient) ──→ 依赖 claudetalk 存活
    │                          ↓ crashes → 断连
    │                          ops-daemon 检测 → 自动重启
    │
    └── 收通知 ──→ 通道 A (feishu-notify MCP) ──→ 不依赖 claudetalk
                    ↑
           SessionEnd hook（提取记忆后通知）
           ops-daemon（告警通知）
           daily-report（日报通知）
```

### 结论

| 场景 | 影响 |
|------|------|
| claudetalk 崩溃 | 手机发消息给 bot 收不到回复（直到 ops-daemon 重启成功） |
| claudetalk 崩溃 | 手机收通知（记忆提取/告警/日报）完全正常 |
| claudetalk 重启 | 重启窗口期（~5-10s）bot 不能收发消息，通知不受影响 |
| claudetalk 多次崩溃触发退避（3600s） | bot 消息通道停 1 小时，通知通道始终正常 |

**"出问题都不会影响手机飞书通讯"——不完全成立。** 只有下行通知（CC→手机）不受影响。上行交互（手机→bot 对话）是绑死在 claudetalk 进程上的。
