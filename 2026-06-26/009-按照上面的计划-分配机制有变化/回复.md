核心变化就三点：

### 1. 从单维判定 → 二维矩阵

```
之前：只看 complexity，只有 moderate 触发
      moderate → hash → 走到 max（qwen3.7-max）或 不走（留pro）

之后：看 complexity × task_type，code/creative/reasoning 各走各路
      moderate+code → hash → kimi-k2.7-code 或 留pro
      complex+reasoning → hash → qwen3.7-max 或 留pro
      moderate+reasoning → 无映射 → 留pro（不触发任何TP）
```

之前 L2 输出 `COMPLEXITY BUDGET` 两词，`task_type` 维度不存在，分配器根本**不知道**这是什么类型的任务，所以只能瞎猜——一律 moderate → max。

### 2. 从二选一 → 5 目标 × 独立阈值

```python
# 之前：一个 bool
should_route_to_max("moderate", req_id, 0.5) → True

# 之后：按 (complexity, type) 查表，返回具体 model_name 或 None
select("moderate", "code",    req_id, 0.5) → "kimi-k2.7-code"  (0.15 < 0.5 → 触发)
select("moderate", "reasoning",req_id, 0.5) → None             (表中无此映射)
select("complex","creative", req_id, 0.5) → "glm-5.2"         (0.15 < 0.5 → 触发)
select("complex","creative", req_id, 0.1) → None              (0.10 < 0.15 → 不触发)
```

每个目标模型有自己的 `min_ratio`：coder 门槛 0.15、max 门槛 0.30、glm 门槛 0.12。余额充足时都开，余额低时贵的先关。

### 3. 改动的实际影响

| | 之前 | 之后 |
|---|---|---|
| 分配模型 | qwen3.7-max 唯一 | kimi-code * glm-5.2 * qwen3.7-max * plus * flash |
| L2 能力 | 不知道任务类型 | 输出 task_type |
| 阈值层数 | 1 个全局 ratio | 5 个 per-model min_ratio |
| moderate+code | 一律走 pro | 分一部分到 kimi-k2.7-code |
| complex+creative | 一律走 max | 分一部分到 glm-5.2 |
| 余额感知 | 所有 moderate 一视同仁 | 每个模型独立退场 |

**不改的：** `trivial`/`simple` 维持 flash 不动，`moderate+reasoning`/`moderate+general` 维持 DeepSeek，`compute_ratio` 的 sqrt 公式不变。

本质是把"一条粗管"换成"五条细管"，各管各的门槛。
