你说得对。之前的双层门槛（全局 ratio + per-model min_ratio）是重复冗余——allocator 已经用 ratio 做了预算感知的 hash 门，分配到模型后还要再卡一次，逻辑上是矛盾的。

应该是：

```
ratio 是唯一大门 → hash 过了就分配，不卡第二道
```

### 简化后的分配机制

```
compute_ratio(rem, total, days)
    ↓ 0.0~0.5
select(complexity, task_type, req_id, ratio)
    ↓ 查矩阵 → 有映射？
    ├─ 无 → None（留 DeepSeek）
    └─ 有 → hash门(ratio) → 过 → 返回 target_model
                              → 不过 → None（留 DeepSeek）
```

**ratio 代表"现在有多大比例走 TP"，分发到具体模型后不再加门槛。**

矩阵简化为纯路由表：

```python
MAPPING = {
    ("moderate", "code"):         "kimi-k2.7-code",
    ("moderate", "creative"):     "glm-5.2",
    ("moderate", "long_context"): "qwen3.6-flash",
    ("complex",  "code"):         "kimi-k2.7-code",
    ("complex",  "creative"):     "glm-5.2",
    ("complex",  "reasoning"):    "qwen3.7-max",
    ("complex",  "long_context"): "qwen3.7-plus",
    ("complex",  "general"):      "qwen3.7-max",
}
```

不在表里的组合 = 不走 TP，留 DeepSeek。

### 单门控制的好处

- `ratio=0.5` → 约 50% 复杂任务走 TP，自动按类型分散到 kimi/glm/max/plus（各取所需）
- `ratio=0.1` → 约 10% 走 TP，各模型等比缩量
- `ratio=0` → 0%，全部流回 DeepSeek

不会再出现"ratio 已经很低了、过了 hash、又被 per-model 阈值拦住"的无效计算。ratio 降下来，所有模型同步退场，不需要单独调每个模型的阈值。

之前浪费在 `qwen3.7-max` 上的 1.4% 流量被自然拆散到 code→kimi、creative→glm、reasoning→max，各自拿到符合自己特长的请求。
