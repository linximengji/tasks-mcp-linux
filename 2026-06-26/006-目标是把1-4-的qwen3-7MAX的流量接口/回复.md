## 修订方案：Multi-Model Allocator

### 核心变化

**当前流水线：**
```
L1/L2判断 → 不可路由 → Allocator (moderate → 50% hash → qwen3.7-max)
```

**改为：**
```
L1/L2判断 → 不可路由 → Allocator (complexity + task_type → 多模型矩阵 → 分层hash)
```

不碰 L1/L2 的主流路由。改动点全部集中在最下游的 Allocator 选择环节。

---

### 1. 扩展 L2 Classifier 输出

当前：`CLASSIFIER_SYSTEM_PROMPT` 输出 `COMPLEXITY BUDGET`（两词）

改为输出 `COMPLEXITY TASK_TYPE BUDGET`（三词）

新增 `task_type` 分类：

| task_type | 判定依据 |
|-----------|---------|
| `code` | 含函数定义、import、API设计、git操作、代码片段 |
| `reasoning` | 架构方案、多步推导、trade-off分析 |
| `creative` | 写作、翻译、润色、总结、文案 |
| `long_context` | 历史消息 > 12K tokens（长文档分析） |
| `general` | 不属于上述 |

L2 classifier 的 prompt 加一句约束：
```
3. Task type: code, reasoning, creative, long_context, or general
```

---

### 2. MultiModelAllocator 映射矩阵

新建 `allocator.py` 替代原有的 `Allocator` 类：

```
(complexity, task_type) → 首选模型 → min_ratio阈值
```

| complexity | task_type | 首选模型 | TP成本 | min_ratio | 理由 |
|-----------|-----------|---------|--------|-----------|------|
| moderate | code | qwen3-coder-plus | ~3x | 0.15 | 代码中等问题，coder 比 max 省 40% |
| moderate | creative | glm-5.1 | ~2x | 0.10 | 中文创作 GLM 强项、价格适中 |
| moderate | reasoning | — | — | — | 留在 DeepSeek，不走 TP |
| moderate | long_context | qwen3.7-plus | 1x | 0.05 | 低成本处理长上下文 |
| moderate | general | — | — | — | 留在 DeepSeek |
| complex | code | qwen3-coder-plus | ~3x | 0.20 | 复杂代码任务 |
| complex | creative | glm-5.1 | ~2x | 0.15 | 复杂中文创作 |
| complex | reasoning | qwen3.7-max | ~5x | 0.30 | 最贵、留给最难的推理 |
| complex | long_context | qwen3.7-max | ~5x | 0.25 | 长文档复杂分析 |
| complex | general | qwen3.7-max | ~5x | 0.30 | 兜底 |

**规则：**
- 表格空白的留在原有 DeepSeek 链路（不触发 TP 消耗）
- `min_ratio` 低于时不触发该模型（hash 概率，平滑过渡）
- `qwen3-coder-plus` 是否 TP 内需确认，如果在 DashScope 按量则另配路由

**GLM-5.1 定位：** 中文创作/润色/长文输出，这个方向它比 Qwen3.7-Plus 更擅长。不做全才，只做 niche。

### 3. @tag 别名清单

```python
_TIERS = {                    # 系统 tier，4 个不变
    "flash":  "deepseek-v4-flash",
    "pro":    "deepseek-v4-pro",
    "max":    "qwen3.7-max",
    "vision": "doubao-1.5-vision-pro",
}

_get_alias() = {              # 扩展别名表
    # task-oriented aliases
    "coder":   "qwen3-coder-plus",    # 代码专用
    "plus":    "qwen3.7-plus",        # 创作 / 翻译
    "cheap":   "qwen3.6-flash",       # 备用按量（非 TP）
    # model-oriented aliases
    "glm":     "glm-5.1",
    "kimi":    "kimi-k2.6",
    "minimax": "MiniMax-M2.5",
    "ds":      "qwen3.7-max-ds",     # DashScope 按量
    "qwen":    "qwen3.7-max",
    "deepseek":"deepseek-v4-pro",
    "doubao":  "doubao-1.5-vision-pro",
}
```

`@glm`、`@kimi`、`@minimax` 都走 TokenPlan 但**仅手工调用**，不加入自动分配。原因：这些模型的专长和 qwen3.7-max 重叠，自动路由的价值不如给用户手动选。

### 4. Bypass 直通 — 无需改动

现有链路覆盖所有场景：

```
X-Proxy-Model: qwen3-coder-plus          → ROUTES[model_name] exact match
@coder 写在消息末尾                       → _fuzzy_resolve_model() → alias → exact
系统 prompt 写死的 model_name             → body["model"] in ROUTES（_route_and_sanitize 第一行）
```

只要在 `litellm_config.yaml` 注册了新模型 + 在 `_get_alias()` 加 alias，bypass 自动生效。无需改 handler 逻辑。

Bypass 场景下不做 budget check——手工强制定向，用户知道自己在干嘛。

### 6. 多模型并行调用 — 方案建议（后续 Phase）

设计草案，不在此次落地：

```
Request:
  POST /v1/messages
  X-Multi-Model: qwen3-coder-plus,glm-5.1  ← 逗号分隔

Response stream（标准 SSE 加 group id）:
  event: message_start
  data: {"group": "coder", ...}

  event: message_start  
  data: {"group": "glm", ...}

  event: content_block_delta
  data: {"group": "coder", "delta": ...}
```

**用 proxylib 的 fallback 模式改造**——fallback 是串行，multi-model 是并行。需要在 model_proxy.py 加一个新的 routing 分支，检测 X-Multi-Model header 后派发到 `_handle_multi_model()`，内部用 `asyncio.gather()` 并发请求多个上游，然后把流式响应交织。

这轮先不碰。等 Phase 1 落地稳定后，用户有实际并行需求时再实现。
