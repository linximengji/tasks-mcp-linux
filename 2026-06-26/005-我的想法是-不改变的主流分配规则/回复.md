## 理解确认

你说的核心是：
1. **L1/L2 主流路由不动** — flash/pro/max/vision 的分类逻辑保持
2. **Allocator 从单模型→多模型** — 不再只是 moderate 哈希概率选 max，而是根据任务特征从 TokenPlan 多个模型中选
3. **价值不在省钱而在多样性** — TokenPlan 模型都比 DeepSeek 贵，但各有专长
4. **@tag 清单要补全**
5. **Bypass 确认是否需要改**
6. **多模型并行调用** — 作为后续功能设计

---

## 设计方案

### 1. 任务类型检测 — 扩展 L2 Classifier

当前 L2 输出：`COMPLEXITY BUDGET`（两词）

改为：`COMPLEXITY TASK_TYPE BUDGET`（三词）

新增 `task_type` 维度：

| task_type | 含义 | 触发特征 |
|-----------|------|----------|
| `code` | 代码生成/调试/重构 | 函数定义、import、API设计、git操作 |
| `reasoning` | 逻辑推理/架构分析 | 多步骤推导、trade-off分析、方案设计 |
| `creative` | 文本创作/润色 | 写作、翻译、改写、总结 |
| `general` | 通用问答 | 不符合上面三类 |

**不改 router.py 的 classify()**，只改 L2 classifier 的 system prompt，多输出一个词。

### 2. Multi-Model Allocator 核心设计

```
当前: moderate → hash → pro 或 max（单维度）
改为: (complexity, task_type) → 多模型选择表 → hash → 选最优TokenPlan模型
```

**分配矩阵（预算充足时的首选）：**

| complexity | code | reasoning | creative | general |
|-----------|------|-----------|----------|---------|
| trivial | (不触发TP) | (不触发TP) | (不触发TP) | (不触发TP) |
| simple | (不触发TP) | (不触发TP) | (不触发TP) | (不触发TP) |
| moderate | qwen3-coder-plus (3x) | 留DeepSeek | qwen3.7-plus (1x) | 留DeepSeek |
| complex | qwen3-coder-plus (3x) | qwen3.7-max (5x) | qwen3.7-plus (1x) | qwen3.7-max (5x) |

**关键改变：** moderate 不再一律走 pro/deepseek，而是 code→coder、creative→plus 走 TokenPlan；complex 也按类型分化。

**预算感知阈值（ratio 低于此值时不触发该模型）：**

| 模型 | 成本倍率 | ratio 阈值 | 理由 |
|------|---------|-----------|------|
| qwen3.7-plus | ~1x | 0.1 | 最便宜，几乎总是开 |
| qwen3-coder-plus | 3x | 0.2 | 中等，余额 40%+才开 |
| qwen3.7-max | 5x | 0.3 | 最贵，余额 60%+才开 |

阈值用 hash 概率实现，平滑过渡，不是一刀切。

### 3. 降级链设计

每个 TokenPlan 模型有自己的降级链：

```
qwen3-coder-plus → (402/429) → deepseek-v4-pro
qwen3.7-plus     → (402/429) → deepseek-v4-pro  
qwen3.7-max      → (402/429) → qwen3.7-max-ds (DashScope 按量)
deepseek-v4-pro  → (402/429) → deepseek-v4-flash
```

这只需要在 litellm_config.yaml 中配置 fallback 字段，handler 现有的 `build_fallback_chain()` 已经支持。

### 4. @tag 别名清单

```python
@flash    → deepseek-v4-flash
@pro      → deepseek-v4-pro
@max      → qwen3.7-max
@plus     → qwen3.7-plus      # 已有，确认
@coder    → qwen3-coder-plus  # 新增
@vision   → doubao-1.5-vision-pro
@ds       → qwen3.7-max-ds    # DashScope
@qwen     → qwen3.7-max
@deepseek → deepseek-v4-pro
@doubao   → doubao-1.5-vision-pro
@cheap    → qwen3.6-flash     # 新增，DashScope按量（非TP）
```

### 5. Bypass 直通 — 不需要改

`X-Proxy-Model` 和 `@model` 语法已经走的是通用查找链路：

```python
# model_proxy.py:216 _fuzzy_resolve_model()
# 1) exact match in ROUTES
# 2) alias via _get_alias()  
# 3) prefix match in ROUTES
```

只要 litellm_config.yaml 里注册了新模型名，bypass 自动生效。`@coder` 和 `@plus` 加好 alias 就行，不需要改 bypass 逻辑。

### 6. 多模型并行调用 — 未来 Phase

设计草案：头 `X-Multi-Model: qwen3-coder-plus,deepseek-v4-pro`，proxy 并行请求两个模型，stream 模式下交错返回每个模型的响应块，标记 `event: model_start(name)` / `event: model_done(name)`。

这个改动量大（handler 全部要改流式逻辑），建议先 Phase 1 落地看效果。
