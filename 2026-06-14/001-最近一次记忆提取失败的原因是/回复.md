原因很清楚。从 `session_hook_debug.log` 最后一条失败记录：

**时间**：18:21-18:22，session `4c865125-41c`

**直接错误**：
```
recall failed: 'utf-8' codec can't decode byte 0xd3 in position 62: invalid continuation byte
FAIL after 23.6s: 'utf-8' codec can't decode byte 0xd3 in position 62: invalid continuation byte
```

**根因**：SessionEnd hook 的 consume 子进程在处理时，LLM 返回了包含非 UTF-8 字节序列（`0xd3`）的内容。在 `recall` 阶段就已经报 UTF-8 decode 失败，LLM 仍然返回了 ACCEPT，但后续处理存储结果时再次触发了同一个 decode 错误。

**不是**最近做的 memory-mcp 合并优化导致的——那个改动在 `_storage.py` 和 `server.py`，存储逻辑本身没问题。这个 failure 是上游 LLM 响应中出现了乱码字节，与 memory_store/update 的 merge 逻辑无关。

属于偶发性的 LLM 输出编码问题，不是代码 bug。
