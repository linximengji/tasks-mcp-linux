## P1：并发消息队列

### 当前问题

```
Msg1 → spawn claude --resume S1 (进程 A)
Msg2 → spawn claude --resume S1 (进程 B)  ← 同时到达，读到同一个 session_id

进程 A 返回 S2，写 sessionMap = S2
进程 B 返回 S3，写 sessionMap = S3  ← 覆盖了 S2
下一条消息用 S3，但 S1→S3 这条分支丢了 Msg1 的回复
```

两个进程 fork 了同一个 session，Msg2 的响应不感知 Msg1 刚说了什么——对话连续性断裂。

### 方案：按 session 串行化

仿照已有的 `compactingPromises` 模式，加一个 `sessionQueues`：

```typescript
const sessionQueues = new Map<string, Promise<void>>()

async function enqueueSession(sessionKey: string): Promise<() => void> {
  let release: () => void
  const prev = sessionQueues.get(sessionKey) || Promise.resolve()
  const next = new Promise<void>(r => { release = r })
  sessionQueues.set(sessionKey, prev.then(() => next))
  await prev  // 等前面的消息处理完
  return release!
}
```

`callClaude` / `callClaudeStreaming` 入口处：

```typescript
const release = await enqueueSession(sessionKey)
try {
  // 原有的 spawn → await → 写回 sessionMap 流程
} finally {
  release()
}
```

效果：

```
时间轴 →
Msg1 → enqueue(S1) → 立即执行 → spawn claude --resume S1
Msg2 → enqueue(S1) → 等待 Msg1 完成 → 这时 sessionMap[S1]=S2 → spawn claude --resume S2 ✓
```

**Side effects**：
- 同一会话的第二条消息延迟 = 前一条的剩余处理时间。群聊中用户 A 和用户 B 同时发消息，如果同在一个 conversation（群聊），会串行。但这是正确的——否则 fork session 丢上下文。
- 不同会话（不同群/不同私聊）不互相阻塞，因为 `sessionKey` 不同。
- `clearedDuringProcessing` 和其他竞态保护可以简化——串行化后不需要了。

### 错误恢复优化

当前：session 失效 → 删 sessionMap → 重建新 session（丢上下文）。

改：`retryCount < maxRetries` 时，**保留原消息重试**。session 失效后重建的新 session 接收相同的消息，用户感知不到上下文断裂。

```typescript
// 当前逻辑（保持）
if (isSessionInvalid) {
  sessionMap.delete(sessionKey)
  saveSessionMap(workDir, sessionMap)
  // 重试：新 session 但相同的消息
  callClaude(options, retryCount + 1).then(resolve).catch(reject)
  return
}
```

这个逻辑已经正确——新 session 不带历史但会收到当前消息。对外表现是"Claude 忘记之前说过什么，但记得当前问题"。要进一步提升需要存历史 transcript（P2 的副产品可以考虑）。

## P2：会话尺寸分层管理

### 当前问题

现有逻辑：`inputTokens > 200K` → async compact。问题：

1. Compact 在回应之后异步跑，用户可能紧接着发下一条消息→继续膨胀
2. 没有硬上限——极端场景可能到 500K+ token
3. Compact 后的 session 仍然累积消息，没有"重置"机制

### 方案：三级阈值

```typescript
const TIERS = [
  { threshold: 200_000, action: 'async-compact' },
  { threshold: 400_000, action: 'sync-compact' },
  { threshold: 600_000, action: 'summarize-reset' },
]
```

#### 第一级（200K）：异步压缩（保持现有行为）

当前 `inputTokens > 200K` → 回复后 spawn compact。不阻塞用户。

#### 第二级（400K）：同步压缩

改 `SessionEntry` 加累计 token 跟踪：

```typescript
interface SessionEntry {
  // ...
  cumulatedInputTokens?: number  // 本 session 累积消费
}
```

当 `cumulatedInputTokens > 400K`，在回应**之前**等待 compact 完成再返回：

```typescript
// buildClaudeArgs 返回后发现需要同步压缩
if (needsSyncCompact && existingSessionId) {
  await compactSession(sessionKey, existingSessionId, workDir, profile, baseArgs)
  // compact 更新了 sessionMap，下一行 buildClaudeArgs 会读到新的 sessionId
}
```

#### 第三级（600K）：自动 summarize + reset

当累积达到 600K：

1. `claude -p --resume {sessionId}` + prompt "Summarize this conversation in 2-3 sentences"
2. 用返回的摘要创建一个**全新的 session**（不 resume）
3. `--append-system-prompt "Previous session summary: {summary}"` 注入摘要
4. 更新 sessionMap 为新的 sessionId，重置 `cumulatedInputTokens = 0`

用户无感知——只收到正常回复。

### 实现细节

**SessionEntry 扩展**：

```typescript
interface SessionEntry {
  sessionId: string
  lastActiveAt: number
  isGroup: boolean
  conversationId: string
  userId: string
  subagentEnabled: boolean
  channel: ChannelType
  cumulatedInputTokens?: number   // 新增
  sessionSummary?: string         // 新增
}
```

**累计 token 计算**（`callClaude` 的回调里已有 `inputTokens`）：

```typescript
if (response.session_id) {
  const entry = sessionMap.get(sessionKey)
  const cumulated = (entry?.cumulatedInputTokens ?? 0) + inputTokens
  // 判断是否需要同步压缩或重置
  if (cumulated > RESET_THRESHOLD && entry?.sessionSummary) {
    // 已经 summarize 过，不再重复——继续使用当前 session
  } else if (cumulated > RESET_THRESHOLD) {
    // 触发 summarize → reset
  } else if (cumulated > FORCE_COMPACT_THRESHOLD) {
    // 下次请求需要同步压缩
  }
  // 更新 entry
}
```

### Side effects

1. **Compact 的额外费用**：每次 compact 消耗一次 API 调用（同等 token 量的输入+输出）。每轮 200K→400K 之间触发一次 async compact + 可能一次 sync compact。按 proxy 路由（DeepSeek v4-pro 等）计算，每次数百到数千 token 的开销。大约每 session 多花 $0.01-0.05。

2. **Summarize 的信息损失**：600K token 压缩为 2-3 句摘要，细粒度上下文（用户提过的具体文件路径、半成品代码、调试过的配置参数）会丢失。这是设计意图——但用户可能注意到"Claude 忘了之前的细节"。

3. **累计 token 准确性问题**：`inputTokens` 来自响应中的 `modelUsage`，但 `/compact` 和 summarize 调用产出的 token 也应该计入累积——否则阈值会偏低。需要把 compact/summarize 的 token 也加进去。

4. **Session 回滚风险**：summarize→reset 后如果用户说"刚才那个方案再说一遍"，新 session 没有细粒度历史可回溯。这是故意折衷——600K 的 session 本身响应也很慢了。

5. **主动对比行**：与 `/new`（用户主动触发的清空）的区别——用户主动 `/new` 是故意丢弃上下文的清空。auto-reset 应该保留摘要，所以行为实质不同。不应影响 `/new` 命令。
