## 改造计划：Step 级智能推送

### 目标

把当前每轮循环中的固定文案（`🧠 深度思考中...` / `🔧 调用工具: Read` / `📋 分析工具结果...`）替换为**自然语言的一步一总结**，让手机端看到的是任务叙事链而非原始状态机。

### 改什么

只改 **一个文件 + 新增一个文件**：

```
src/index.ts                ← 修改流式回调，改 editStatus 调用逻辑
src/core/step-summarizer.ts ← 新增：工具名+入参/结果 → 一句话总结
```

### Step 1：新增 `step-summarizer.ts`

一个纯函数，输入 `(toolName, input)` → 输出人类可读的步骤描述。类似：

| 工具 | 输入示例 | 输出 |
|---|---|---|
| `Read` | `{file_path: "src/index.ts"}` | `📖 读取 src/index.ts` |
| `Write` | `{file_path: "src/config.ts"}` | `✏️ 写入 src/config.ts` |
| `Edit` | `{file_path: "src/index.ts"}` | `📝 修改 src/index.ts` |
| `Bash` | `{command: "npm test"}` | `💻 执行: npm test`（截断长命令到 40 字符） |
| `Glob` | `{pattern: "src/**/*.ts"}` | `🔍 搜索 src/**/*.ts` |
| `Grep` | `{pattern: "editStatus"}` | `🔎 搜索 'editStatus'` |
| `Agent` | `{description: "Explore...", prompt: "..."}` | `🤖 子任务: Explore push logic`（取 description 或 prompt 前 30 字） |
| `WebSearch` | `{query: "..."}` | `🌐 搜索: {query 前 30 字}` |
| 未知 | — | `🔧 {toolName}` |

同时提供 `summarizeToolResult(toolName, resultPreview)` → 结果总结：

| 工具 | 输出 |
|---|---|
| `Read` | `已读取 {path}（{lines} 行）` |
| `Write/Edit` | `已保存 {path}` |
| `Bash` | `命令退出码 {code}`（成功时不显示） |
| `Glob/Grep` | `找到 {n} 个结果` |
| `Agent` | `子任务完成 ✓` |

### Step 2：修改 `index.ts` 流式回调

核心思路：**从"状态通知"切到"步骤叙事"**。

改动点：

1. **步骤计数器**：加 `let stepNumber = 0`，每次 `tool_use` 事件自增
2. **`thinking` 事件**：不再每次都推。只在满足以下条件时推：
   - 这是第一个 thinking（还没任何 step）
   - 或上次推送超过 5 秒没更新（兜底保活）
   - 文案改为 `⏳ 正在处理...` 而非 `🧠 深度思考中...`
3. **`tool_use` 事件**：调用 `summarizeToolCall(event.toolName, event.toolInput)` → `editStatus(summary)`。例如 `📖 步骤 1/6: 读取 src/index.ts`
4. **`tool_result` 事件**：调用 `summarizeToolResult(event.toolName, event.toolResult)` → `editStatus(resultSummary)`。例如 `📖 步骤 1/6: 已读取 src/index.ts (312 行) ✓`
5. **总步数估算**：维护 `totalSteps` 变量，每次 tool_use 时 `totalSteps = Math.max(totalSteps, stepNumber + 3)`（预留余量），这样显示 `步骤 1/5` 而不是 `步骤 1/?`
6. **连续相同 step 文案去重**：保留现有的 `text === lastEditText` 检查即可

### Step 3：考虑 20 次编辑限制

飞书单条消息最多 edit 20 次。当前代码已有兜底（`EDIT_LIMIT_REACHED` 时发新消息）。改造后：
- 每个 step 会 push 2 次（tool_use + tool_result），10 个 step 就用完配额
- **方案**：合并 tool_use + tool_result 为一条消息，在 tool_result 到达时只更新一次。标记 `pendingToolUse`，延迟到 tool_result 或下一个 tool_use 时才实际 edit

```
tool_use → 存到 pendingStep，不实际 edit
tool_result → editStatus("📖 步骤 1: 已读取 src/index.ts (312 行)")
下一个 tool_use → editStatus("🔧 步骤 2: 搜索 editStatus 调用处")
```

这样每个 step 只占 1 次 edit，20 个 step 也能覆盖。
