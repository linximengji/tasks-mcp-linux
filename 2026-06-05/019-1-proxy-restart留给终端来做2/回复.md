---

## 当前链路

```
飞书消息 → claudetalk
  → 查 .claudetalk-sessions.json 找 session_id
  → claude -p --resume {session_id}  (stdin 发消息)
  → proxy /v1/messages
  → router 分配模型
  → API 响应
  → 提取新 session_id 写回 sessions.json
  → 飞书回复
```

会话核心依赖 **Claude CLI 内部 state**（`~/.claude/sessions/`），claudetalk 只存一个 opaque `session_id` 字符串。

---

## 五个连续对话的薄弱环节

### 1. 模型路由跳跃 → 历史格式污染

最关键的连续性 bug：

| 轮次 | 用户消息 | Router 路由 | 响应包含 |
|------|---------|-------------|---------|
| 1 | "帮我设计一个架构" | → v4-pro | 正常 text |
| 2 | "考虑这个 edge case" | → v4-pro | 正常 |
| 3 | "用图表解释" | → qwen3.6-maas | 正常 |
| 4 | "继续" | → flash (trivial) | 正常 |

**问题**：Claude CLI 内部累计的历史消息中，assistant 消息格式可能包含模型特定的东西。Claude CLI 从它自己的 state 重建完整 messages 数组发给 proxy，这些消息的 `content` 数组排列、特殊 block type 可能只在某些模型上生产过。

sanitizer 虽然修了 `thinking`/`redacted_thinking`，但问题不限于此——还有 `tool_reference`（已移除）、以及不同模型对 `tool_use`/`tool_result` 的表示差异。

### 2. 单次调用失败 → 上下文不对齐

```
用户: "帮我写个排序算法"
  → claude -p --resume sess1
  → proxy → v4-pro → 超时 或 400
  → claudetalk 返回错误给用户
  → Claude CLI session 还在，但标记了异常
用户: "继续"
  → claude -p --resume sess1
  → Claude CLI 恢复 session，但可能重发上次的消息，或处于"等待回复"的中间状态
```

当前错误处理只有重试 2 次（`claude.ts:571-587`），两次都失败就直接暴露错误。**没有"安全回滚"或"重新同步"机制**。

### 3. Session 跨重启丢失

`claude` CLI 重启后内部 session state 可能会清理。claudetalk 进程重启后从 `sessions.json` 读取的 `session_id` 可能已失效。现有代码处理了 `"No conversation found"` 错误并自动重建，但代价是**用户丢失历史上下文**。

### 4. 并发消息竞争

```
用户 A 发消息 → claude -p --resume sess1 (进程 1)
用户 A 很快又发一条 → claude -p --resume sess1 (进程 2)

进程 1 返回，写 sessionMap[sess1] = newId1
进程 2 返回，写 sessionMap[sess1] = newId2  ← 覆盖了进程 1 的结果
```

下一条消息用 `newId2`，但 `newId1` 的那轮对话的回复从未被发送。实际场景：飞书群聊中两条消息几乎同时到达。Claude CLI 的 session 是**线性**的，不是并发安全的。

### 5. Token 积累无上限

`/compact` 在 200K token 时触发，但：
- compact 本身也走一次完整调用（耗时、花钱）
- compact 后的新 session 包含 compact prompt + 回复，上下文实际上继续膨胀
- 没有**硬上限**——一条超长对话可能达到 1M token，延迟和费用都不可控

---

### 总结：需要解决的三个优先级

1. **P0**：`thinking` 等专有 block 跨模型泄漏 → **已修**（sanitizer 层）
2. **P1**：并发消息竞争 + 失败后上下文对齐 → 需要 claudetalk 层加锁和重试策略
3. **P2**：会话尺寸管理 → 引入主动压缩 / 轮换策略
