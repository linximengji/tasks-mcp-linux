对，私聊和群聊的 `chat_id` 不同，所以 `getSessionKey(conversationId, ...)` 天然就是两个不同的 session，互不干扰。

所以问题在于：同一场景下（比如私聊），一个 conversationId 只有一个 session。这才是 `/fork` 的切入点。

我前面列的 `/tasks`、`/compact`、`/summary` 确实太平了。只说真正有分量的：

---

### **`/bg` — 后台任务**

当前架构限制：`message handler` 里 `await callClaude()` 是串行的，必须等它返回才能处理下一条消息。一旦有耗时任务（重构、大文件分析），手机端就卡死，发啥都不响应。

`/bg` 就是把发过来的消息 fork 出去跑，handler 立即返回"已提交后台任务"，这样手机端可以继续聊天。后台跑完了通过飞书 API 主动推送结果回来。

**实现要点**：
- 剥离成一个独立进程（spawn child process，传 message + sessionId）
- 飞书 channel 的 `sendMessage` 需要在 global 级别可调用（当前只有 channel 实例持有）
- 不需要额外的 worker 池——单条后台任务 spawn 一个 claude CLI 进程即可

### **`/commit` / `/diff` / `/log` — Git 操作链**

这是手机端最痛的：改了代码想提交，但干不了。一套 git 指令可以把手机端变成"轻量代码管理器"：

```
手机: /diff          → 看改了什么
手机: /commit "xxx"  → 提交
手机: /push          → 推送
手机: /log -5        → 看最近提交
手机: /branch        → 看分支
```

纯 claude CLI 调 git，不涉及 session 管理，实现上就是截获消息前缀走 spawn git 而非 callClaude。

### **`/fork` / `/switch` — 分支会话**

本质是在 `sessionMap` 上多加一层 key 映射：

```
当前:
  chatId "oc_xxx" → sessionId "abc"
/fork 后:
  chatId "oc_xxx" → sessionId "abc" (主)
  chatId "oc_xxx" + fork: "B" → sessionId "def" (分支)
  currentFork: "B"
/switch A 后:
  currentFork: null (主)
```

改动量不大（`getSessionKey` 多加一个 fork name 维度），但手机端体验提升明显——查个资料不用怕打断主会话的上下文。

---

这三个里，`/commit` 实现成本最低，`/bg` 用户体感提升最大，`/fork` 架构最优雅。你倾向哪个方向？
