三个瓶颈，逐个拆：

**1. Knowledge 标签太粗（全标 `toolchain`）**

| 方案 | 思路 | 成本 | 效果 |
|------|------|------|------|
| A. 开 LLM | `--skip-llm` 去掉，让 knowledge/memory 走 deepseek-v4-flash 提炼 | 每条 session ~200 tokens，147 条约 30K tokens ≈ ¥0.03 | 标签区分度大幅提升，能从对话中提取真正意图 |
| B. 细化 heuristic | 扩充关键词库，分更多类（排查问题/写代码/配置修改/研究调研/闲聊测试） | 零 | 比现在好但天花板低——关键词永远 cover 不了自然语言多样性 |
| C. 混合 | heuristic 粗筛 → LLM 只处理 top 20% 高价值 session（长对话、多工具调用） | 约 1/5 的 A 方案成本 | 兼顾成本和区分度 |

建议：**C 混合**。日常跑 heuristic 打底 + LLM 精炼高质量 session，每天新增 session 不多（<5 条），成本可忽略。

**2. Memory 太噪（120KB，147 条 verbatim）**

当前问题：短测试 session（2-3 turn、0 tool call）和长任务 session 混在一起，`user_ask_summary` 就是原文照抄。

| 方案 | 思路 |
|------|------|
| A. 加信号门槛 | turns < 3 或 tool_calls < 2 的 session 直接跳过，不写 memory |
| B. LLM 压缩 | 把同一天/同类 session 喂给 LLM 做批量摘要，输出 `{日期, 主题, 关键决策, 产出}` |
| C. 聚类后摘要 | 先按工具使用模式聚类 → 每类取代表 session LLM 摘要 |

建议：**A+B 组合**。先过滤低信号 session（预计砍掉 60%+），剩下的用 LLM 压缩成日报式摘要，memory 文件从 120KB → 10-20KB，可读性大幅提升。

**3. Patterns 全是 0（无意义）**

当前只统计了 error_count，但 CC 的 jsonl 里 error 本身就是 0。这个维度没价值。

方案：
- 换维度：统计**工具调用序列模式**（如 `Read→Edit→Bash` 三联）、**session 时长分布**、**用户打断频率**、**任务完成率**（最后一条 assistant 消息是否包含 tool_use）
- 或者直接砍掉这个 phase，等有明确需求再加
