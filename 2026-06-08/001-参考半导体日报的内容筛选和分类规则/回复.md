好，全貌清楚了。当前问题根源：

**筛选** — 关键词匹配title 1个或summary 2个，关键词覆盖有限，相关文章可能因措辞不同被漏掉，不相关文章可能因恰含关键词被收入。

**分类** — 一行一行 `text.includes(kw)`，匹配到第一个类别就停止，无法处理多主题交叉的文章，也无法判断"这篇文章到底在讲什么"。

以下是方案：

---

## 方案：LLM 筛选 + 分类，替代关键词逻辑

### 总体思路

保留 RSS 采集的原始条目获取（去掉关键词过滤），引入一段 LLM 调用同时完成两件事：**这篇文章是否值得收录**（相关性判断） + **分到哪个类别**。

### 具体改动

现有管线：

```
fetchAll()  ← 关键词过滤
  → translateBatch()
  → classifyNews()  ← 关键词分类
  → renderHtml()
```

改为：

```
fetchAllRaw()  ← 只取原始条目，不做关键词过滤
  → filterAndClassifyByLLM()  ← 一次 LLM 调用完成筛选+分类+摘要
  → renderHtml()
```

**fetch.ts**：新增 `fetchAllRaw()`（或给 `fetchOne` 加参数跳过关键词过滤），只保留去重和排除无关标题（`EXCLUDE_TITLE_KEYWORDS`）。

**新增 `llm.ts`**：核心逻辑，一次批量调用 DeepSeek 完成三件事：

```
输入：每篇文章的 title + summary（原始英文/中文混合）
输出：[序号] 保留/丢弃 | 类别 | 中文标题(可选) | 中文摘要(≤80字)
```

Prompt 设计要点：
- 类别为当前 7 类（含"其他"），多加一个"丢弃"标记
- "丢弃"用于：完全不相关、软文/广告、泛科技非半导体
- 对每篇文章给置信度标签（high / medium / low），low 的归入"其他"而不丢弃，保持 recall
- 类别匹配规则：语义匹配而非关键词匹配，支持交叉归属（但每篇只选一个最匹配的）

**classify.ts**：可以删掉或大幅简化，弃用 `CLASSIFICATION_KEYWORDS` 映射。

### 调用时机

放在 `translateBatch` **之后**（这样 LLM 可以看到中文翻译，判断更准）：或者合并翻译和分类为一次调用以省 token。推荐**合并方案**：

```
原始英文文章
  → 一次 LLM 调用：翻译 + 筛选 + 分类
  输出：[序号] 保留/丢弃 | 类别 | 中文标题 | 中文摘要(≤80字)
```

这样把当前的 translate + humanize + classify 三步 LLM 调用压缩为一步。

### Token 成本估算

以日均 150 条原始条目、每条约 200 token 输入计：
- 输入：150 × 200 ≈ 30K token
- 输出：150 × 50 ≈ 7.5K token
- 单次调用，约 40K token / 天
- DeepSeek API 成本约 ¥0.5/天，完全可以接受

### 收益

- 不再维护 600+ 行的关键词配置（`config.ts` 中大量关键词）
- 分类准确率大幅提升
- 减少"其他"类中遗漏的好文章
- 管线从 3 次 LLM 调用减为 1 次

---

### 需要你确认的点

1. **合并方案**还是**分离方案**？即将翻译+筛选+分类合并为一次 LLM 调用，还是拆成两次（翻译一次，筛选+分类一次）？
2. 当前关键词配置文件（`config.ts` 中 `RSS_FEEDS` 里的 keywords、`CLASSIFICATION_KEYWORDS`、`HIGHLIGHT_KEYWORDS`）——是否保留一个精简版本做粗筛去噪，还是完全交给 LLM？
3. 日均文章量大概在什么范围（我看目前最多 80 条/源 × 9 源 = 720 条，实际进入管线的很少）？这影响是否需要在 fetch 阶段先做粗筛。
