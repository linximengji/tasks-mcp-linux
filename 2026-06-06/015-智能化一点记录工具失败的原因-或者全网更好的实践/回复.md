## 生产实践

现有社区工具和框架的做法更实用：

**1. 错误分类 + 分级响应**

不只是"失败了几次"，而是**失败类型**：

| 分类 | 检测信号 | 策略 |
|---|---|---|
| Tool Failure | stderr, exit != 0 | 可重试（带 backoff） |
| Syntax/Parse Error | 报错里含语法相关关键词 | 警告后阻断 |
| Permission/IO Error | 权限/文件不存在 | 直接阻断，重试没意义 |
| Circular Reasoning | A→B→A→B 交替重复 | 注入 hint 迫使换方向 |
| Stuck Agent | N 步无输出 | 硬阻断 |

**2. Agent Guard MCP** — circuit breaker + pattern detection（`exact_repeat` / `low_diversity` / 组合），MCP 协议

**3. Noos (Rust)** — scope drift 检测 + cost circuit break + procedural correction memory，"5+ consecutive same tool without interleaving" 触发

**4. cc-discipline (npm)** — Claude Code 专用，shell hook 阻断 5 次内连续编辑同一文件

**5. 三层终止架构** — Step Counter → Error Accumulation Counter → Goal Proximity Check（每步问"这步真的在靠近目标吗？"）

**6. ReLoop** — Python 库，failure memory graph + vector search，语义匹配当前错误到历史解法

## 针对我们场景的最佳选择

对于 hook 场景（~100ms 耗时，Python 脚本，读 stdin JSON），**力所能及的"智能"升级**：

### 1. 错误类型分类（从 stderr/stdout 解析）

```python
# 简单的关键词匹配，不调 LLM
ERROR_PATTERNS = {
    "syntax":        r"(SyntaxError|Unexpected token|parse error|语法错误)",
    "type_error":    r"(TypeError|cannot be|no attribute|类型错误)",
    "file_not_found": r"(ENOENT|No such file|not found|找不到)",
    "permission":    r"(EACCES|Permission denied|权限)",
    "network":       r"(ECONNRESET|ETIMEOUT|connect.*refused|超时)",
    "empty_result":  r"(No matches|empty|没有结果|0 lines?|not found)",
}
```

### 2. 签名升级：tool + error_type + semantic_input

不再只是 `sha256(tool+input)`，而是：
- Edit：`sha256(error_type + file_path + old_string)` → 同文件同旧代码但**不同错误类型**=不同签名
- Bash/PS：`sha256(error_type + command[:200])`
- 接口调用：`sha256(error_type + api_name + arg_hash)`

这样第一次是 `SyntaxError`，第二次是 `FileNotFound` → 不命中签名 → 不会误判为重复失败。

### 3. 失败模式趋势检测

同一 `error_type` 在 30 分钟内出现 3+ 次（不同 input）→ 不是输入问题了，是**思路有问题**。这比"相同 input"的检测更高一层。

### 4. 成功即清零

一次成功执行 → 清空该 tool 的所有错误历史。这是所有生产系统一致的做法，避免了"历史错误阻碍正常操作"。

---

**相比纯哈希方案的升级**：

| 维度 | 纯哈希 | 加分类 + 趋势 |
|---|---|---|
| 误杀率 | 中（分步改文件被误认为重试） | 低（不同错误类型不视为重复） |
| 检测率 | 高（精确匹配） | 中（模式匹配有漏） |
| 附加信息 | 无 | 能统计"哪些错误类型最常见" |
| 复杂度 | 30 行 | ~80 行 |
| 开销 | ~5ms | ~10ms |
