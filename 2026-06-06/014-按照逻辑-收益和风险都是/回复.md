## 收益

**直接收益**

1. **打断死循环** — CC 最常见的浪费时间模式：同一个 Edit 反复改不对 → 再试 → 还是不对。阻断能直接止损，省 token。

2. **对抗注意力衰减** — 长对话里早期错误沉到上下文深处，模型"忘记"了。`systemMessage` 是硬注入，不依赖模型从历史里检索。

3. **0 人工干预** — 不需要人盯着 terminal 喊停，尤其飞书远程场景下（人不在电脑前）有价值。

**附带收益**

4. **失败模式可审计** — `failure-memory.jsonl` 积累数据后，能看出"哪个工具/哪种操作最容易触发重试"，反向优化 prompt 或换方案。

---

## 风险

**1. 误杀（核心风险）**

PreToolUse 的签名匹配是纯规则，没有语义判断。以下场景会误判：

- 同一文件确实需要多次 Edit（分步修改不同区域），但 `file_path` 匹配 = 误判为重复失败
- 同一命令确实需要重试（网络波动、API 偶尔超时），不是逻辑错误

**缓解**：`file_path` 做 Edit 签名太粗糙。应该用 `file_path + old_string` 做 Edit 签名，这样只阻断"对**完全相同**的旧代码反复尝试修改"。但即使如此，如果 CC 第一次 `old_string` 写错 → 第二次修正了 `old_string` → 第三次又回到错的版本——这就检测不到。反之如果 CC 坚持用同一个 `old_string`，确实说明没在想新方案。

**2. 退出码阻断的副作用**

PreToolUse exit non-zero（阻断）时，CC 看到的是"工具调用被拒绝"，可能产生两种反应：
- 好：被迫换思路
- 坏：CC 不理解为什么被拒，开始用更奇怪的 workaround（比如换个工具做同样的事）

**缓解**：阻断时要给明确的 `systemMessage` 说明原因和替代建议。但不保证 CC 会读。

**3. 时间窗口硬编码**

我说的是"30 分钟内"，但实际场景：
- 真的修好了，30 分钟后还想改同一个文件 → 签名还在 window 内，误判
- 错误本质上没过期，但 30 分钟到了，签名过期 → 循环重来

**缓解**：不用固定时间窗口。用"连续失败次数"：第 1 次警告，第 3 次阻断，**成功一次就清零**所有签名。这样只有真的在循环重试时才会触发，恢复就放行。

**4. 签名存储膨胀**

每次失败写一行 JSONL，如果不清理，长期运行会积累大量无价值数据。

**缓解**：成功一次后清理对应签名（不是全清，只清该文件/命令的条目）。另外定期 trim 超过 24h 的。

---

## 权衡结论

| | 收益 | 风险 |
|---|---|---|
| 断死循环 | 省 token，止损 | 误杀概率中等，需精细签名 |
| systemMessage 注入 | 可靠打断注意力 | CC 可能无视 |
| 全自动 | 免人工 | 阻断后可能产生新错误模式 |

**最大风险是 Edit 签名设计**。如果签名太粗糙（只用 file_path），正常分步编辑也会被阻断，体验极差。如果用 `file_path + old_string` 精确匹配，误杀率可接受——因为"对完全相同的旧代码反复尝试"本身就是 bug 循环的特征，不是正常行为。

要不要实现？如果做，我会用：签名 = `file_path + old_string`（Edit）/ `command`（Bash/PS），成功即清零，第 3 次阻断。
