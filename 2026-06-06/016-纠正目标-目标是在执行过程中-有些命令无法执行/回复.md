明白了。问题不是"同一个 session 内反复重试"，而是**跨 session、跨项目的无效命令重复尝试**。

典型场景：
- Windows 上没有 `apt-get`，但 CC 每次新项目都试着跑
- PowerShell 5.1 不支持某语法，CC 还是会写
- 某个二进制根本没装，下次照样试

核心需求：**持久化的系统能力黑名单**，让 CC 记住"这类命令在这个环境上永远跑不通"。

---

## 方案：跨项目错误记忆

### 存储模型

```json
// ~/.claude/failure_memory.json
{
  "binary_not_found": {
    "apt-get":    {"first_seen": "2026-06-01", "count": 12, "block": true},
    "brew":       {"first_seen": "2026-05-20", "count": 8,  "block": true},
    "systemctl":  {"first_seen": "2026-06-03", "count": 3,  "block": false}
  },
  "ps_syntax": {
    "&&":         {"first_seen": "2026-05-15", "count": 20, "block": true}
  }
}
```

### 签名策略：按"不可用的东西"匹配，而非按完整命令

| 失败类型 | 提取签名 | 示例 |
|---|---|---|
| `command not found` | 二进制名 | `apt-get install vim` → `apt-get` |
| PS 语法错误 | 语法模式 | `npm run build && npm test` → `&&` |
| 模块/包不存在 | 模块名 | `import foo` → `foo` |
| 文件永久不存在 | 文件路径模式 | `/usr/bin/xxx` → `/usr/bin/` |

关键区别：`apt-get install vim` 和 `apt-get update` 都匹配到 `apt-get`——因为是**工具不可用**，不是参数问题。

### Hook 流程

```
PreToolUse:
  1. 解析命令，提取候选签名（二进制名、语法模式等）
  2. 查 failure_memory.json
  3. 命中且 block=true → exit 2 阻断，systemMessage 告知"此工具在此环境不可用"
  4. 命中但 block=false → systemMessage 警告，放行

PostToolUseFailure:
  1. 解析 stderr，判断失败类型
  2. 提取签名，写入/更新 failure_memory.json
  3. 同签名累计 ≥3 次 → 自动升级 block=true
```

### 为什么比纯哈希好

- **跨命令泛化**：`apt-get install vim` 失败后，`apt-get update` 也被拦截
- **跨项目共享**：记忆文件在 `~/.claude/`，所有项目共用
- **渐进升级**：第一次警告，第三次才阻断，给"万一用户装了新工具"留余地
- **可人工清理**：删文件或编辑 JSON 即可重置
