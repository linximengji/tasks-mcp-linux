## 深度分析：awesome-mcp-servers 对我们的价值

### 先看我们现在有什么

当前 `claude.json` 里只注册了 **3 个 MCP Server**：

| 现有 | 类型 | 覆盖领域 |
|------|------|---------|
| p1s-printer | 硬件控制 | 3D 打印机 |
| blender-mcp | 创意工具 | 3D 建模 |
| image-gen | 创意工具 | AI 图像生成 |

全是**垂直专用**工具。缺少通用基础设施层的 MCP Server。

### 2026 MCP 生态的真实规模

- **22,000+** 个 MCP Server 已注册（mcp.so）
- MCP SDK 月下载量 **9,700 万次**
- 头部 50 个 MCP Server 月搜索量 **62.2 万次**
- 实际使用模式：开发者**叠加 4-8 个** MCP Server，一条 prompt 完成跨工具链操作

### 对我们有价值的 MCP Server（按 ROI 排序）

#### Tier 1 — 立刻装，显著提升能力

| MCP Server | 解决什么问题 | 为什么值 |
|------------|------------|---------|
| **Playwright** (Microsoft 官方) | 浏览器自动化、网页测试、截图 | 82k/月搜索量，CC 已有插件。调试前端、抓动态网页、截图验证，全场景覆盖 |
| **Context7** | 实时库文档查询 | 32k/月搜索。写代码时不用切出去查最新 API，直接让 CC 拉取文档 |
| **Firecrawl** | 深度网页抓取 + 结构化提取 | 7.2k/月搜索。对 semi-report 管线可能比 RSS 更精准 |
| **Sequential Thinking** | 复杂多步推理链 | 13k/月搜索。让 CC 的深度推理有结构化框架 |
| **Memory/Knowledge Graph** | 持久化记忆 | 记住跨会话的上下文，不需要每次重新描述项目 |

### 价值判断

**awesome-mcp-servers 非常有价值，但不是因为它有 22,000 个 server。而是因为它解决了"MCP 没有内置发现机制"这个协议缺陷。**

具体对我们：

1. **补齐基础设施短板** — 我们现在只有 3 个垂直专用 server，缺 Playwright 做浏览器交互、缺 Context7 做实时文档、缺 Memory 做跨会话记忆。这些都是通用能力，装一个就提升整个 workspace 的上限。

2. **Composability 是真正的杠杆** — 单看每个 MCP Server 都很小，但叠加后产生组合爆炸。比如："用 Playwright 打开 OctoEverywhere，截图我的 P1S 状态，通过 claudetalk 发到飞书"——这需要 3 个 MCP Server 协同。

3. **但不要掉进"装一堆"的陷阱** — 22,000 个 server 里 95% 跟我们无关。精选 5-8 个就够了。MCP Server 的本质是**工具的 API 化**，每多一个就是多一个 token 消耗。按需加载比全量预装好。

### 建议下一步

先装 Tier 1 的 3-5 个（Playwright + Context7 + Sequential Thinking + Memory + Firecrawl），然后实际用一周感受下组合能力的提升，再决定是否追加。

要我直接开始装 Playwright MCP 和 Context7？这两个在 CC 官方插件市场里，一条命令就能装上。
