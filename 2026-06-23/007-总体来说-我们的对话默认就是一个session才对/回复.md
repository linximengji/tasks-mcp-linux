## Resume 失败的核心机制

在 `_execClaude` (`src/core/claude.ts:754-926`)，resume 失败后的回退逻辑有一个 **关键漏洞**：

### 正常路径
```
消息1 → buildClaudeArgs → 无 sessionId → 新建 session → 返回 sessionId=A
消息2 → buildClaudeArgs → 有 sessionId=A → --resume A → 正常回复 → sessionId=B
```

### Resume 失败路径
```
消息2 → --resume A → claude 进程 exit code != 0
  → 检测 stderr 含 "No conversation found" / "session ID" / "Invalid session"
  → 从 sessionMap 删除该条目 (line 800)
  → 读取 deadSummary (line 798-799)
     └─ 只有之前触发过 summarizeAndReset (>600k tokens) 的 session 才有
  → 若有 summary → 注入到 processedMessage → 重试 (新 session)
  → 若无 summary → **重试时零上下文** → 用户看到"忘记你是谁"
```

这就是根本原因。`sessionSummary` 只在超过 `RESET_THRESHOLD=600k` 时才写入，正常短期的 session 被 invalidate 后，fresh session 没有任何旧对话信息。

### 什么导致 resume 失败

1. **claudetalk 进程重启后会话过期** — restarted claudetalk 从 `.claudetalk-sessions.json` 读出旧的 `sessionId`，但 Claude CLI 的 session 文件可能已被清理或属于上一个进程实例

2. **异步 compact 竞态** (`claude.ts:896-914`) — 消息1 完成后触发异步 compact，更新 sessionId。消息2 到达时如果 compact 未完成，`pendingCompact` 会阻塞等待。但 compact 完成后新消息才拿到 old sessionId 构建 args，此时 compact 已在后台更新了 sessionId — 这取决于具体的时序

3. **exit code=0 但 JSON 解析失败也可能丢 session** — line 916: `catch (parseError)` 命中时 resolve 原始 stdout，不会保存新的 sessionId。但如果 `response.session_id` 不存在，line 861 的 if 分支不执行，sessionMap 保留旧 sessionId，下次还能再试

### 判断"resume fail 是主因"的证据

看这段日志模式（来自日志解析，`index.ts:618`）：`"No conversation found"` 或 `"session ID"` 只要出现过一次，那个 conversation 后续所有消息都会走"new session"路径，用户感知就是"每次都忘记之前说了什么"。

想确认的话可以查日志：

```bash
grep -E "Session invalid|No conversation found|Invalid session" D:/ClaudeProjects/.claudetalk/claudetalk-*.log
```

### 修复方向

如果确认是这个问题，最简单的修复是在 session 删除前 **将最后 N 轮对话缓存为 summary**，而不是只依赖 ≥600k 才有的 `sessionSummary`。比如每次成功回复后，在 `SessionEntry` 里保留最近 2 条消息的摘要。需要我修吗？
