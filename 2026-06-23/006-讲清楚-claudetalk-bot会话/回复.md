## 两个系统的本质区别

### 1. claudetalk bot 会话 — 后台常驻服务

claudetalk 是一个**独立的后台 Node.js 进程**（由 ops-daemon 管理生命周期），与当前你在 CLI 中交互的 Claude Code 会话**完全无关**。

**架构链路：**
```
手机飞书发消息 → 飞书 WebSocket 长连接
  → claudetalk 进程 (node cli.js) 接收
  → 判断是否为内置指令(/status /new /restart 等)
  → 否则 spawn 子进程 claude -p --resume <session_id> 处理
  → 捕获 stdout → 回复飞书群/私聊
```

关键是：每次你手机发消息给飞书机器人，claudetalk 会**新开一个 Claude CLI 子进程**来处理该消息，不是直接使用你现在终端里这个 CC 会话。

**会话管理：**
- 每个飞书 `conversationId`（群聊或私聊的 ID）映射到一个独立的 Claude CLI session
- Session key 格式：`conversationId\x00workDir\x00profile\x00channel`
- 不同 channel（feishu / dingtalk / discord）的 session **完全隔离**
- 持久化在 `.claudetalk-sessions.json`
- 支持 `/new` 清空、`/session list` 查看、`/session name` 命名
- 三级 token 阈值管理（200k/400k/600k 自动 compact）

### 2. feishu-remote skill — 管理界面（不是消息通道）

`feishu-remote` 是一个 Claude Code 的 skill，它**在当前 CC 会话中运行**，提供的是对 claudetalk 后台服务的管理命令：

```
你: "启动飞书远程" → 激活 feishu-remote skill
  → skill 运行 start/stop/status/setup 命令
  → 管理 claudetalk 进程的启停
```

它本质上就是 claudetalk 的**遥控器**，不是消息通道本身。

### 3. Cloudflare Tunnel "远程终端" — 完全不同的东西

除了上述两个，还有第三条路径：
- 你在飞书机器人发"开启远程"
- claudetalk 识别到 TERMINAL_OPEN 指令
- 启动 `cloudflared tunnel` + `ops-dashboard`（端口 8765）
- 浏览器访问 `https://term.linximengji.com` → ops-dashboard Web UI
- 这个 Web UI 是**真正的远程终端**，可以在浏览器里直接操作这一台 CC

### 总结对比

| 维度 | claudetalk bot 会话 | feishu-remote skill | Cloudflare 远程终端 |
|------|-------------------|-------------------|-------------------|
| 进程 | 独立后台 node 进程 | 当前 CC 会话内 | 独立 cloudflared 进程 |
| 用途 | 飞书↔CC 桥接 | 管理 claudetalk | 浏览器远程操作 CC |
| 消息来源 | 飞书群/私聊 | 当前 CC 输入 | 浏览器 Web UI |
| 会话类型 | claude -p 子会话 | 当前 CC 主会话 | 当前 CC 主会话 |
| 会话持久化 | `.claudetalk-sessions.json` | CC 内置 session | CC 内置 session |

**最核心的认知：** 你在手机上飞书发消息时，claudetalk 会以 `--resume` 模式启动多个独立的 `claude` CLI 子进程来处理，这和当前你面前这个 CC 会话是两个独立的 Claude 实例。当前 CC 看不到飞书那边的对话历史，飞书那边也看不到你当前 CC 的内容——它们各管各的 session 空间。
