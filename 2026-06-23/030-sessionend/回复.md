## 问题定位

管线分两层过滤，两层的盲区：

**Stage 1 — `_should_stage`（无 LLM，hook 内）**
```
n >= 3 and (tools > 0 or has_anchor) → stage
```
条件太宽松。一次调研对比（2-3 轮对话，调了 WebSearch）就够触发。

**Stage 2 — `fmt_prompt`（LLM curator）**
REJECT 列表只写了 `env setup, tutorial, chitchat, status check, test run, incomplete debug, tool exploration, playing around` —— 没有 "research comparison"、"information lookup"、"news inquiry"。所以 LLM 收到一篇有工具调用、有内容的调研，很自然会认为这是知识，写入记忆。

核心矛盾：**管线设计是"能提取就提取"，而你需要的策略是"只有架构/功能/决策类才值得存"。**

## 推荐方案

三层联动，不绕路：

**1. `_should_stage` 加信号强度评分（行 115-133）**

不是二元的 `stage / skip`，而是算一个 `signal_score`，低于阈值跳过 staging：
- 有文件编辑 (tool_use with file_edit) → +3
- 有 git commit/push → +3
- 有 anchor 关键词（修 bug/根因）→ +2
- 纯 WebSearch/RawSearch → -1
- 只有 1 轮用户提问 → -2

```python
def _signal_score(turns):
    score = 0
    for t in turns:
        if t["has_anchor"]: score += 2
        if t.get("has_edit"): score += 3   # 新增标记
        if t.get("tool_count", 0) > 0:
            # 检查 tool 类型... 需要从 transcript 解析
            pass
    # 惩罚单轮查询类
    if len(turns) <= 2: score -= 2
    return score
```

门槛：`>= 2` 才写 staging。

**2. 纯查询类直接打 `research` 标签跳过 curate**

staging 文件加字段 `skip_curate: true`，`curate_staging` 遇到直接删除不调 LLM。

**3. LLM curator prompt 加 reject 规则**

即使过了前两层，prompt 里补一条拒绝标准：
> REJECT if: pure information lookup, model comparison, news inquiry, research summary with no code/config change, feature investigation without implementation.
