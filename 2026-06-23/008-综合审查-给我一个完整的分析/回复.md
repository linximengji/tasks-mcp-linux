# claudetalk Session Resume 失效全面审查

**日期**：2026-06-23  
**范围**：`D:\ClaudeProjects\claudetalk\src\core\claude.ts`（1220 行）
**审查方式**：逐行代码分析 + 并发模型推演
**现象**：手机飞书发消息，claudetalk 回复不记得上一条对话的内容

---

## 一、Session 架构总览

### 1.1 数据流

```
飞书消息 → FeishuClient.handleMessageEventAsync()
  → channel.onMessage handler (index.ts:396)
  → callClaude / callClaudeStreaming
  → enqueueSession (per-sessionKey 串行化)
  → _execClaude / _execClaudeStreaming
  → buildClaudeArgs() → 构造 --resume sessionId 参数
  → spawn("claude", ["-p", "--resume", sessionId, ...])
  → 解析 JSON 响应 → 提取新 sessionId → 落盘
```

### 1.2 Session 持久化

```
文件: {workDir}/.claudetalk-sessions.json
Key:   conversationId\x00workDir\x00profile\x00channel
Value: SessionEntry { sessionId, cumulatedInputTokens, sessionSummary, ... }
```

Session key 包含 4 段：`conversationID` + `workDir` + `profile` + `channelType`。不同 channel 的 session **完全隔离**。

### 1.3 三级 Token 阈值管理

| 阈值 | 行为 |
|------|------|
| 200k (ASYNC_COMPACT_THRESHOLD) | 响应后异步触发 `/compact` |
| 400k (SYNC_COMPACT_THRESHOLD) | 请求前同步等待 `/compact` 完成 |
| 600k (RESET_THRESHOLD) | 请求前 `summarizeAndReset`，清空 sessionId，保留 sessionSummary |

## 二、主根因链条：为什么飞书对话不记得上一条

### 2.1 Resume 失败 → 零上下文回退

```
消息2 → buildClaudeArgs
  → sessionMap.get(sessionKey) 存在 → 取 sessionId=SID_A
  → args = ["-p", "--resume", "SID_A", ...]

消息2 → spawn claude --resume SID_A

  — exit ≠ 0, stderr 含 "No conversation found" —

  → isSessionInvalid=true (line 791)
  → sessionMap.delete(sessionKey)                     (line 800)
  → deadSummary = deadEntry?.sessionSummary           (line 799)
  → sessionSummary 只在 ≥600k 后 summarizeAndReset  才会写入！
  → 若无 deadSummary → 重试时 processedMessage 无注入 (line 803-804)
  → retry → 新 session（零上下文）
```

**这是你看到的"不记得上一条"的精确代码路径。**

`sessionSummary` 字段（line 35）在整个设计中的角色是"600k 阈值溢出后的逃生舱"，而不是"常规 resume 失败的降级策略"。这意味着：**99% 的正常对话 session 都没有 sessionSummary**，一但 resume 失败就什么上下文都拿不回来。

### 2.2 什么导致 resume 失败

claudetalk 的所有 session 数据完全依赖 claudetalk 进程自身维护的 `.claudetalk-sessions.json` 文件中的 `sessionId`。当以下情况发生时，该 sessionId 会指向一个 Claude CLI 不认识的 session：

| 场景 | 概率 | 说明 |
|------|------|------|
| claudetalk 进程重启 | 高 | Claude CLI 的 session 是进程级缓存，进程重启后旧 sessionId 失效 |
| 文件写竞争导致 sessionId 回退或丢失 | 中 | 见第三章 Bug #3 / #4 |
| 异步 compact 更新 sessionId 后，claude 子进程返回旧 sessionId | 中 | 时序竞争 |
| session 文件损坏 | 低 | Windows 崩溃时 atomic write 缺失 |

## 三、并发模型审查：8 个独立 Bug

### Bug #1 — **Session-invalid retry 硬编码 channel 为 `dingtalk`**（P0）

**位置**：`claude.ts:708, 806, 1169`

```typescript
// line 708: destructuring 默认值
channel = 'dingtalk',

// line 806: retry 时 channel 被硬编码覆盖
_execClaude({ ...options, channel }, retryCount + 1)
```

飞书通道传入 `channel='feishu'`。retry 时 `channel` 被覆写为 `dingtalk`。而 session key 构造包含 channel 段：

| session | channel | key |
|---------|---------|-----|
| 原始 | feishu | `oc_xxx\x00D:\ClaudeProjects\x00\x00feishu` |
| retry | dingtalk | `oc_xxx\x00D:\ClaudeProjects\x00\x00dingtalk` |

**后果**：retry 后的 session 存在 `dingtalk` key 下。下次消息按 `feishu` 查不到，每次都新建空 session。**终身的 resume 失败。**

`_execClaudeStreaming`（line 1169）同款 Bug。

**修复**：删掉 destructuring 默认值，直接用 `options.channel ?? 'dingtalk'` 或者在 retry 中不要覆盖 channel。

### Bug #2 — **`sessionSummary` 缺失导致 retry 无上下文注入**（P0）

**位置**：`claude.ts:798-804, 1162-1168`

```typescript
const deadEntry = sessionMap.get(sessionKey)
const deadSummary = deadEntry?.sessionSummary       // 多数情况是 undefined
sessionMap.delete(sessionKey)
if (deadSummary) {                                   // 几乎永远不成立
  options = { ...options, processedMessage: `[Previous conversation summary: ${deadSummary}]...` }
}
```

`sessionSummary` 只在触发 `summarizeAndReset`（≥600k input tokens）时才写入。正常短期对话 `sessionSummary` 恒为 `undefined`。

**改进方案 A（轻量）**：每次成功响应后，在 `SessionEntry` 保存 `lastMessage` + `lastReply`，retry 时注入。

**改进方案 B（完整）**：将 `sessionSummary` 改为类似 [SQLite WAL 模式] 的增量日志，每次工具调用完自动追加一句话摘要。

### Bug #3 — **`loadSessionMap` 静默吞异常 → 全量 session 丢失**（P1）

**位置**：`claude.ts:69`

```typescript
try {
  const content = readFileSync(sessionFile, 'utf-8')
  const raw = JSON.parse(content)
  // ...
} catch (error) {
  log(`[session] Failed to load sessions: ${error}`)
  return new Map()  // ← 静默返回空！所有已保存 session 丢失
}
```

Windows 崩溃/断电时，`writeFileSync` 写入一半的文件导致 JSON 解析异常 → `catch` 命中 → 返回空 Map。紧接着 `saveSessionMap`（line 879 等）覆盖写回空 `{}`，**不可逆损坏**。

**修复**：`saveSessionMap` 使用 atomic write：先写临时文件 → `renameSync` 覆盖原文件。

### Bug #4 — **Compact + Message 的 sessionId 写入竞态**（P1）

**位置**：`claude.ts:460-466` vs `claude.ts:860-883`

`enqueueSession`（line 406）按 sessionKey 串行化消息处理，但 **compact 在消息的 Promise 链之外**：

```
时序轴：
消息A → enqueueSession → _execClaude → 响应 → resolve() → 触发 async compact (line 910)
                                                                    ↓ compact 完成
                                                                    sessionMap.set(sessionId=C)
消息B → enqueueSession → _execClaude → buildClaudeArgs → 读到 sessionId=C → ✓ resume C
        ↓
消息B → 响应 → sessionMap.set(sessionId=B)                ← 覆盖回 B！
        ↓
                                                                              saveSessionMap → C 被覆盖
```

反过来也成立：compact 先完成，消息 B 先响应，compact 的 `saveSessionMap` 覆盖 B 的 sessionId。

**根因**：`compactingPromises`（line 398）只防止并发 compact，不保证 compact 的 `sessionId` 写入与消息响应的 `sessionId` 写入不冲突。两者写的是同一个 `sessionKey`。

**修复**：在 `enqueueSession` 的队列尾部追加 compact 的 sessionId 更新操作，使其在队列序列内闭环。

### Bug #5 — **`preProcessSessionCheck` 后重建 args，但 channel 可能已被抹掉**（P2）

**位置**：`claude.ts:727-735`

```typescript
const rebuilt = buildClaudeArgs(options)    // options 中有 channel
args = rebuilt.args
existingSessionId = rebuilt.existingSessionId
```

`buildClaudeArgs` 内部（line 641）调用 `getSessionKey(options.conversationId, ..., options.channel)`。如果 `options.channel` 是 `undefined`，会回退到默认行为。这条实际受 Bug #1 的保护墙——workDir+conversationId 匹配时大概率拿到同一 entry——但不干净。

### Bug #6 — **Exit code = 0 但 sessionId 为空/缺失 → session 静默丢失**（P2）

**位置**：`claude.ts:860-883`

```typescript
if (response.session_id) {   // ← falsy → 不执行这个块
  // 更新 sessionMap ...
}
```

Claude CLI 返回 `{"result": "...", "session_id": ""}` 或 `{"result": "..."}`（无 session_id 字段）。此时不更新 sessionMap，旧的 `sessionId` 保留。但如果 `existingSessionId` 原本就是 undefined（全新会话），那这条消息之后**没有任何 session 被保存**。用户发下一条消息时，`buildClaudeArgs` 找不到 sessionId → 再次新建 → 再次无上下文。

### Bug #7 — **`wasClearedDuringProcessing` 的 delete 副作用**（P2）

**位置**：`claude.ts:137-142`

```typescript
function wasClearedDuringProcessing(...): boolean {
  return clearedDuringProcessing.delete(sessionKey)  // ← delete = 消费标志
}
```

`wasClearedDuringProcessing` 调用一次就把标志消费掉了。而它在两个地方被调用：
1. `compactSession`（line 463）— 检查是否需要写入 compact 后的 sessionId
2. `_execClaude` 响应处理器（line 862）— 检查是否需要保存响应 sessionId

一个 `clearSession` 调用（line 132）设置标志 → compact 的 `wasClearedDuringProcessing` 消费掉 → 响应处理器的检查返回 `false` → 认为"没有被清空" → 把旧 sessionId 写回去。

**后果**：`/new` 清空后，如果 compact 刚好在运行，响应处理器可能把一个过期的 sessionId 覆盖写入。

---

### Bug #8 — **无 TTL/过期清理**（P3）

**位置**：全局

每个对话、每个 profile、每个 channel 的组合都在 `.claudetalk-sessions.json` 中生成一个永久条目。只追加不删除。虽有 `lastActiveAt` 字段但无任何读取方用它做过期判断。

## 四、Streaming vs Non-Streaming 路径差异

| 维度 | 非流式 `_execClaude` | 流式 `_execClaudeStreaming` |
|------|---------------------|--------------------------|
| isSessionInvalid 检测 | 含 `'--resume'` | 不含 `'--resume'`（漏一条） |
| sessionId 保存时机 | 在 close handler 中，解析 JSON 后 | 在 `result` stream event 中，**及** close handler 兜底 |
| 失败降级 | 无降级 | subagent 配置变化时降级为非流式 |
| retry channel 硬编码 | 同款 Bug #1 | 同款 Bug #1 |

## 六、结论

**你 100% 猜对了。** "手机端不记得上一条对话"的根本原因就是 resume 失败，且失败后的 retry 路径有两条致命缺陷：

1. **Bug #1**：channel 硬编码导致 retry 后的 session 存到错误 key 下，后续消息永远找不到它
2. **Bug #2**：sessionSummary 在 99% 的正常对话中都是 undefined，retry 时无上下文注入，等同于全新开始

修复这两个 P0 即可解决用户可感知的问题。Bug #3（atomic write）和 Bug #4（compact 竞态）是潜在的数据损坏源头，建议一并修掉。
