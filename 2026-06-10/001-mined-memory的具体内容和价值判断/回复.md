## Mined Memory 内容与价值分析

### 数据全貌

```
172 sessions 评分 → 95 LLM 摘要 → 87 写入磁盘
                                    ├─ 81 global（跨项目）
                                    ├─ 4 claudetalk
                                    ├─ 1 ops-daemon
                                    └─ 1 p1s-mcp
```

### 评分高度依赖活动量，不反映知识价值

看两组典型案例的对比：

| Session | Score | Turns | Tools | 实际内容 |
|---------|-------|-------|-------|---------|
| 7f59b145 | **10** | 30 | 150 | "检查claudetalk运行状态与watchdog守护进程，确认日志及守护均正常" |
| 222f0fd4 | 5 | 9 | 44 | "整理了D盘项目文件并分析冗余内容，确定了需要清理的文件清单" |
| a4accbd8 | 6 | 14 | 122 | "测试会话，验证基础工具可用性，无特定任务产出" |

Score=10 的系统健康检查 vs Score=5 的实际生产力任务 — 评分完全颠倒。

---

### 三类内容的实际价值

#### 第一类：有用 — 结构化的活动历史（~40 条，score 8-10 中真正有产出的）

```
session-b004ebf9 (9): Token花费统计与报表系统，设计了模型主动路由切换策略
session-a68effdc (8): 用MCP工具改造semi report输出格式，确保链接有效性
session-cab164bb (9): 本地大模型部署方案，含硬件采购与NAS存储
session-859a159d (8): Task Scheduler中semi-report异常，脚本路径错误修正
```

作用：帮你回忆"某天做了什么事"、"某个决策是在什么上下文中作出的"。

**但每条只是一个标题** — 没有方案细节、没有关键参数、没有代码变更摘要。

#### 第二类：半有用 — 故障记录但缺少根因（~25 条，score 6-7）

```
session-a3d7a891 (7): "排查ops daemon守护失败，根因是进程状态检测逻辑异常并修复"
session-768817b1 (8): "排查ClaudeTalk服务异常，网络请求测试与日志分析定位连接故障"
session-b72bedaa (6): "手机任务失败，根因为执行环境配置错误，通过调整参数解决"
```

记录了"出过什么问题"但没有记录"怎么修的"—— 改了哪个文件的哪一行、什么配置参数。下次同类故障发生时，这些记忆提供了检索入口但无法直接复用修复方案。

#### 第三类：纯噪音（~22 条，score 5-6 的低价值 + score 8-10 的健康检查）

```
score=5: "常规系统测试，验证Bash与文件编辑等核心工具" (session-76c5bfdb)
score=10: "检查claudetalk运行状态与watchdog守护进程，确认日志及守护均正常" (session-7f59b145)
score=8: "无明显任务目标，可能进行了随机的工具探索或环境检查" (session-05f218f2)
score=6: "测试会话，验证基础工具可用性，无特定任务产出" (session-a4accbd8)
```

高 score 的健康检查混进来了 — 它们在评分公式里拿高分，但内容是对系统的例行确认，纯日志性质。

---

### 深层问题：Score 公式是相关性，不是因果性

```
当前 Score ∝ turns × tool_calls × unique_tools × duration × thinking_pct
理想 Score ∝ 知识增量（决策、踩坑、问题根因、配置变更、API 发现）
```

两条管道里的高分 session 对比：

| | 7f59b145 (score=10) | 7601eee7 (score=9) |
|---|---|---|
| Turns | 30 | **81** |
| Tools | 150 | **613** |
| 内容 | "确认服务正常" | "全面分析WorkBuddy继承项目，指出模块耦合" |
| 知识价值 | 零（活动日志） | 高（架构分析） |

7601eee7 的 81 turns + 613 tool calls 产生了真实的架构分析知识，值得保留。7f59b145 的 30 turns + 150 tool calls 只是服务状态确认，却拿了最高分。

---

### 结论

| 维度 | 状态 |
|------|------|
| 覆盖度 | 87 条，172 中覆盖 50%，留下 76 条低分未写入 |
| 信息密度 | 每条 1 句摘要 + metrics，无 actionable knowledge |
| 可检索性 | MEMORY.md 索引 + LLM description，关键词可搜到 |
| 去重 | 无 — 相同类型的检查/调试会留下多条语义重复记录 |
| 衰减 | 无 — 纯噪音的健康检查与架构决策平等存在 |

**当前 mined memory = 一个会自动过期的会话历史标题索引，不是知识库。**

能回答"5 月 29 日我做了什么"，不能回答"上次 proxy 路由出问题是怎么修的"。

需要第二层抽取吗？方案：
1. 在 `digest_to_memory.py` 里加一个 "价值分类器" — 检测到 debug/fix/design 等关键词时标记为 `actionable`，例行检查标记为 `activity_log`
2. actionale 类型的 session 额外调一次 LLM 追问 "具体改了什么、为什么这样改"
3. activity_log 类型的 session 降低 confidence 或自动过期清理
