调查结论如下：

## ops-daemon 问题的根因链

### 直接原因：OTLP exporter 阻塞启动

`main.py` 顶部（第 7-19 行）在导入阶段就初始化 OpenTelemetry：

```python
os.environ.setdefault("OTEL_EXPORTER_OTLP_ENDPOINT", "http://localhost:4317")
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter(insecure=True)))
```

当 **Jaeger (port 4317) 未启动**时，`BatchSpanProcessor` 的后台线程持续重试连接，**阻塞在模块加载阶段**。daemon 进程启动但永远进不了 `run()` 主循环。

### 连锁反应

1. **PID 锁独占** — 卡死的 daemon 通过 `msvcrt.lockf()` 持有 `daemon.pid` 的写入锁
2. **新实例被拒** — 后续启动的 daemon 检测到锁 → 输出 "another daemon holds the PID lock" → 退出
3. **protected_pids.json 自愈循环** — 每次新 daemon 启动会注册自己的 PID 到 protected_pids，导致 `protect_services.py` hook 拦截后续的 taskkill 操作
4. **僵尸累积** — 最终发现多个 daemon 进程（14244、23100、10564 等）全部卡在 OTLP 初始化

### 暴露的系统性问题

| 问题 | 影响 |
|------|------|
| OTLP 启动失败无超时退路 | Jaeger 关闭时 daemon 永远无法启动 |
| PID 锁 + protected_pids 形成死锁 | 卡死进程杀不掉（hook 拦截），正确的进程起不来 |
| daemon 不自愈 | 没有启动超时检测或自救机制 |
| `_cloudflared_auto_stop` 也依赖 OTLP | daemon 不可用意味着自动 stop tunnel 也失效 |
