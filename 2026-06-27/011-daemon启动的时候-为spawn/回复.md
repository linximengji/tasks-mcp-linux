## 答案：daemon 确实没有在启动时直接 spawn claudetalk

代码 `main.py:547-550` 明确写了：

```python
# Boot-time child process spawning intentionally omitted.
# All services (proxy, claudetalk, MCP server) are covered by the
# period check-repair cycle (every 60s).
```

daemon 启动后第一个 check 周期（~60s 后）检测 claudetalk 状态：
- 读 `claudetalk-default.pid` → 查 tasklist → PID 活着 → 返回 `"status": "up"` → **不 repair**
- 逻辑是对的

## 三个实例的根因

**核心 bug 不在 daemon 启动，在 restart 流程有 overlap 窗口**：

```
spawn_claudetalk(kill_first=True) 流程：
  1. stop_claudetalk()
     → _kill_by_pid_file() → _kill_pid()
       → os.kill(pid, SIGTERM)     ← 不等待，立即返回
                                ← 此时旧进程还在 drain（最长 60s）
  2. subprocess.Popen(...)         ← 新进程启动
  3. 写 PID 文件 = 新 PID          ← 只记录了最新一个

结果：
  旧进程（draining中） + 新进程 → 两个实例共存
  再 restart 一次 → 旧旧 + 旧 + 新 → 三个实例
```

`_kill_pid()` 发 SIGTERM 后**不等待进程退出**就立刻 spawn 新的。而 claudetalk 的 `drainThenExit` 最多等 60s（LLM 调用完成）。这 60s 窗口里两个实例同时运行，共享同一个飞书 WebSocket 连接。

## 用户看到"daemon 启动时 spawn claudetalk"的原因

实际链条是：

```
daemon 被 TS 杀 (14:20)
→ 旧 daemon 的 claudetalk (PID 22016) 存活
→ 新 daemon 启动 (14:20:23)
→ check 发现 PID 22016 活着 → 不 repair
→ ... 期间有人触发 restart（如飞书 /restart）
→ stop_claudetalk 发 SIGTERM 但不等待
→ 新实例 (PID 4764) 启动，旧 (22016) 还在 drain
→ 再触发一次 → 第三个实例 (PID 6552)
→ daemon 的 check 只看 PID 文件（6552）→ 显示"up"

用户视角：daemon 启动后 claudetalk 被重启了
实际原因：daemon 没 restart claudetalk，是 restart 机制的 SIGTERM-no-wait 导致重叠实例
```
