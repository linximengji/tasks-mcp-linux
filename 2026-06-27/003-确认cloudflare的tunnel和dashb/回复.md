确认完毕。**ops-daemon 没有对这两个服务的自动修复**：

- **cloudflared** — 有 `check_cloudflared` + `_cloudflared_auto_stop`，但**只会在 degraded 超过 600s 后停止隧道，不会重启**。没有修复，只有上限。且只检查连接数，不检查进程存活。
- **ops-dashboard** — 零保护。没有 check、没有监控、没有自动修复。

## 修复方案

涉及 2 个文件，3 处改动：

### Fix 1：`tunnel_manager.stop()` 加日志（根因）

**问题**：`_kill_port()` 整个 try/except/pass；`stop()` 的 taskkill 结果也不写日志。失败不可追踪。

**改动**：
- `_kill_port()` 日志记录 `kill 了哪些 PID / 没找到 / 失败`
- `stop()` 记录 `taskkill /IM` 的 stdout/stderr
- 所有异常写入 episodic 日志

### Fix 2：`claudetalk` 关闭远程的等待策略

**问题**：PS1 内先 stop 4 个 services（含 docker stop），然后才 kill cloudflared。docker stop 耗掉几秒后，claudetalk 侧 `waitForTunnelGone(15000)` 15s 不够覆盖 Windows 进程清理的尾巴。

**改动**：`stop()` 内先杀 cloudflared，再停 services，减少 cloudflared 从 kill 到检查的时间差：

```
# 之前
for svc in services: _stop_service(svc)   # docker stop 耗时
taskkill /F /IM cloudflared.exe
# 之后
taskkill /F /IM cloudflared.exe          # 先杀 tunnel
time.sleep(1)
for svc in services: _stop_service(svc)   # 再停服务
```

同时小幅增大 claudetalk 的 `waitForTunnelGone` 超时从 15s → 25s。

### Fix 3（可选）：claudetalk 检查 PS1 exit code

**问题**：`exec` callback 只看 `err` 参数，不看退出码。PS1 执行失败会完全静默。

**改动**：检查 `stderr` 内容和非零退出码，失败时记录日志。
