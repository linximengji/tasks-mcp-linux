## 完整分析

从 exit-marker.json 和源码追到了飞书通知的完整链路。

---

### 通知源头：两条独立路径

**路径 A — claudetalk 自身上线通知（index.ts:381-384）**
```
_trySendCrashNotification("claudetalk 异常退出")
```
通过 claudetalk bot 自身的飞书 channel 发私聊消息。触发点：
- `SIGINT/SIGTERM` 信号 → `drainThenExit()` 发送 `"signal=SIGTERM"`
- `unhandledRejection` → 发送 `"unhandledRejection: <错误详情>"`
- `beforeExit` → 写 exit-marker 但不发通知

**路径 B — ops-daemon 发 CRITICAL 卡片（main.py:378-412）**
```
notify("CRITICAL", "claudetalk 快速崩溃超过阈值", "...退避 Xs 后自动重试")
```
通过 ops-daemon 的 `notify.py` → 飞书 tenant API 发彩色卡片。触发条件：15s 内崩溃 ≥5 次。

---

### 本次事件的根因链路

从 `exit-marker.json` 可知：

| 字段 | 值 |
|------|-----|
| `startup_phases` | `["initLogFile", "loadConfig"]` |
| `uptime_s` | **0**（进程启动瞬间就退出）|
| `code` | **1** |
| `stack` | `cli.js:778:13`（`main().catch(...)`) |

启动到 `index.ts:317` 处 `loadConfig(workDir, profile)` 返回 `null` → `throw new Error("找不到配置")` → 被 cli.js:776 的 `main().catch()` 捕获 → `process.exit(1)`。

**可能原因**：ops-daemon 的 `process_manager.py:193` spawn claudetalk 时指定的是 `--profile default`，但 `.claudetalk.json` 中工作目录下没有名为 `"default"` 的 profile（或者配置文件 JSON 损坏/路径不匹配）。

完整触发链：
```
claudetalk loadConfig失败 → exit(1) → 
ops-daemon 检测到 crash marker →
spawn_claudetalk(kill_first=True) 重试 →
依然 loadConfig 失败 → 再次 crash →
... 重复 5 次（均在 15s 内）→
_claudetalk_lifecycle 触发：
  "claudetalk 快速崩溃超过阈值，退避 300s" →
  notify_feishu → 飞书 CRITICAL 卡片 → 手机通知
```

---

### 雪上加霜：ops-daemon 自身不稳定

`ops-daemon-start.log` 显示 ops-daemon 也在频繁被重启：
```
01:30   heartbeat stale → daemon restart
02:10   heartbeat stale → daemon restart
14:20   heartbeat stale → daemon restart
14:40   heartbeat stale → daemon restart
```

`startup_err.log` 显示根因是 **Jaeger/OTLP 端口 4317 不可达**，导致 OpenTelemetry `BatchSpanProcessor` 卡住无法写入新 trace → heartbeat 超时 → Task Scheduler 判定死亡 → kill daemon。daemon 重启后重新 spawn claudetalk，但 claudetalk 的 loadConfig 故障未修复，再次 crash → 恶性循环。

### 用户手机收到多条通知的来源

| 消息内容 | 发送方 | 触发条件 |
|----------|--------|----------|
| `"[Crash] claudetalk 异常退出，原因: signal=SIGTERM"` | claudetalk 自身飞书 channel | daemon 重启时发 SIGTERM 给旧进程 |
| `"[CRITICAL] claudetalk 快速崩溃超过阈值"` | ops-daemon notify.py | 15s 内 crash ≥5 次 |
| `"[CRITICAL] ops-daemon: claudetalk repair failed"` | ops-daemon notify.py | 修复次数耗尽（`repair_max=2`） |
| `"[CRITICAL] ops-daemon crashed"` | ops-daemon notify.py | daemon 自身因 OTLP 卡死被 Task Scheduler 杀死 |

### 修复方向

两个问题叠加，需要解决优先级最高的那个：

1. **Jaeger/OTLP 4317 不可达** → daemon heartbeat 被阻塞 → daemon 反复重启 → 连带 claudetalk 被反复 spawn → 每次 spawn 都触发 loadConfig 失败 → 快速崩溃循环
2. **claudetalk loadConfig 失败** — 需检查 `D:\ClaudeProjects\.claudetalk.json` 中是否有 `"default"` profile
