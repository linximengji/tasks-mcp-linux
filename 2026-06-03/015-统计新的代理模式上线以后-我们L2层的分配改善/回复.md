## L2 层效果分析（新代理上线后，当前 session 331 请求）

### L1 vs L2 分流

| 层级 | 请求数 | 占比 |
|---|---|---|
| L1 快速规则 | 319 | 96.4% |
| L2 Flash 分类器 | 12 | 3.6% |

L1 完全主导。`recent-tools` 规则一条就拦截了 90%（298 次），因为工具调用是 CC 最常见的消息模式。

### L2 分类结果

12 次全部 `moderate + ? → deepseek-v4-pro`。**复杂度永远 moderate，预算永远未知。**

| 本应出现的模型 | 实际出现 | 次数 |
|---|---|---|
| flash (trivial/simple) | 0 | — |
| pro (moderate) | 12 | ✅ |
| qwen3.7-max (complex) | 0 | — |

### 预算调节

**零次触发**。原因两层：
1. `budget_est` 永远是 `?` → 没法判断 low/medium/high
2. `prefer_level = 0.7` 落在不调节区间 (0.3~0.8)

### 根因：`max_tokens=20` 被 thinking 耗尽

DeepSeek 开启 thinking 时，thinking token 计入 max_tokens。20 tokens 只够输出一个 `moderate`，第二个词（`medium`）永远来不及：

```
thinking: "We need to classify this as moderate..."  ← 已消耗 15 tokens
text: "moderate"                                     ← 只剩 5 tokens，第二个词不够了
```

### Token 统计也有问题

| 模型 | 请求 | input tokens | output tokens |
|---|---|---|---|
| deepseek-v4-pro | 310 | 5,005,382 | 149,683 |
| deepseek-v4-flash | 3 | 135,277 | 409 |
| qwen3.7-max | 8 | **未记录** | **未记录** |
| qwen3.6-plus | 10 | **未记录** | **未记录** |

qwen 模型的 token 统计丢失——它们走 Anthropic 协议但 `record_tokens` 应该写在通用路径里，可能是 provider 判断逻辑漏了。

### 结论

L2 分类器目前完全无效：每请求 200ms 延迟 + 50 token 消耗，12 次全部输出相同结果，零次预算调节触发。同时 qwen 模型的 token 统计有遗漏。两个都是 bug，不是配置问题。
