## 模型断连调查结果

### 概述

代理日志显示两类断连：**6/2 晚间 DeepSeek 间歇性不可用** + **多 Provider 同时瘫痪事件**。

### 6/2 22:00 — 多 Provider 同时瘫痪（最严重）

```
22:00:22  qwen3.6-maas   → ConnectionResetError → fallback 豆包
22:00:22  豆包            → URLError → 无 fallback
22:00:23  deepseek-v4-pro → degraded（被熔断）→ fallback qwen3.7-max
22:00:23  qwen3.7-max    → URLError → 无 fallback
```

**所有 4 个 Provider 同时挂**。这不是单一服务故障——要么是本地网络断了，要么是 proxy 自身异常（之后出现了 10054 强行断开和 10048 端口占用）。

### 6/2 22:41-22:45 — DeepSeek 熔断期

```
22:41:18  deepseek-v4-pro → 400  → fallback qwen3.7-max
22:41:30  deepseek-v4-pro → 400  → fallback qwen3.7-max  （连续 3 次 400）
22:41:53  deepseek-v4-pro → degraded（熔断） 跳过
...连续 7 次 degraded 直到 22:45
```

DeepSeek 返回 HTTP 400（非 5xx），按熔断规则 3 次失败触发 5 分钟熔断，之后 5 分钟内所有请求直接走 fallback。

### 根因分析

| 时间段 | 故障 | 根因推测 |
|--------|------|---------|
| 22:00 | 4 provider 同时不可达 | **本机网络闪断** — URLError + ConnectionReset 同时出现，不是单一远端的特征 |
| 22:41 | DeepSeek 返回 400 | DeepSeek 服务端异常，返回格式错的响应。不是网络问题，是服务内部 |
| 23:30 | DeepSeek ConnectError + 端口 10048 | **proxy 重启**后旧进程未释放端口 |
| 21:58 | DeepSeek + qwen 双 ConnectError + 10054 | 远程主机强行关闭连接 — 代理层拦截或 TLS 握手失败 |

### 已知问题

1. **熔断判定条件太宽** — HTTP 400（请求格式错误）不应该触发熔断，这是上游服务自己的 bug，不是网络问题。400 ≠ 服务不可用。
2. **端口 10048 错误** — proxy 非正常重启时，端口 4000 未释放，新进程起不来。watchdog 应该有端口冲突检测但可能没触发。
3. **10054 强制关闭** — ConnectionResetError 出现频率较高，可能是本地安全软件或路由器 TCP keepalive 策略导致长连接被掐断。

### 影响

从日志看，fallback 链在大多数情况下能成功路由到可用模型，但 22:00 的那次全挂持续了约 12 秒没有可用模型（4 provider 同时 URLError/ConnectionReset），期间请求直接失败。
