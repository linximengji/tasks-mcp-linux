对，你的判断很准。当前 watchdog 的 `is_port_open()` 只做 TCP connect 探测，覆盖的故障面很窄：

## 当前覆盖 vs 实际故障面

| 故障场景 | TCP connect 能发现？ | 实际影响 |
|---|---|---|
| 进程崩溃 / OOM 被杀 | ✅ port 断开 | 代理完全不可用 |
| 进程挂起（死锁/无限循环） | ❌ port 仍可连 | 所有请求卡住，无响应 |
| 上游 API 全部不可达 | ❌ port 仍可连 | 每个请求都 502/error |
| 所有路由 degraded | ❌ port 仍可连 | fallback 链耗尽的 fallback 链…最终还是 502 |
| Python GIL 卡死（C扩展阻塞） | ❌ port 仍可连 | 同挂起 |

port 检测只能抓到进程级别的崩溃，而代理真正需要切备份的场景往往是**进程活着但不可用**。

## 现状：internal telemetry 已经有了

`telemetry.py` 里有相对完善的内部追踪：

- `route_health` — 单模型连续 3 次失败 → degraded，300s 恢复窗口
- `stats["errors"]` / `stats["errors_by_model"]` — 全局错误计数
- `stats["latency_ms"]` — 平均延迟
- `is_degraded()` — 判断单路由是否健康

问题是 **watchdog 完全没读这些**。它只看 port。

## 改进方向

最小改动——watchdog 做 HTTP health check 而非 TCP connect：

```python
# 当前：只看 port
def is_port_open(port):
    s.connect_ex(("127.0.0.1", port)) == 0

# 改为：实际发 HTTP 请求，超时检测
def proxy_healthy(port):
    try:
        r = urllib.request.urlopen(f"http://127.0.0.1:{port}/health", timeout=3)
        return r.status == 200
    except Exception:
        return False
```

这能覆盖**进程挂起**场景，但还覆盖不了"上游全挂"——因为 `/health` 现在是静态 `{"status":"ok"}`，不反映内部状态。

再进一步——让 `/health` 返回真实健康信息，watchdog 做语义判断：

```python
# /health 返回
{"status":"ok", "healthy_routes":3, "degraded_routes":2, "error_rate":0.15}

# watchdog 判断
if healthy_routes == 0 or error_rate > 0.8:
    switch_to_backup()
```

---

要不要做？我个人建议分两级：
1. **立即**：watchdog 改 HTTP check + timeout（覆盖挂起），改动 3 行
2. **后续**：`/health` 返回内部状态 + watchdog 做语义判断（覆盖上游全挂）
