## 问题本质

给 AI 说"加个搜索功能"，AI 不追问"搜什么字段？分页？权限？排序？"，直接选一个它认为最合理的方案。假设错了就是 **Silent Failure**——代码看着对，实际不符合真实需求。学术界 ReqElicitGym（arXiv 2026）实测：LLM 自主发现隐性需求的覆盖率 < 50%。

## 社区三类方案

### 一、Prompt 级（零成本，立刻可用）

**Assumption Inventory Prompt**（dev.to 2025）—— 在写代码前发给 AI 的 preamble：
1. 用 1-2 句复述目标（验证理解一致性）
2. 列出至少 6 个假设，分 Product/UX、Data/Edge Cases、Security、Performance 四组，标注 Low/Medium/High 风险
3. 最多 5 个澄清问题（防分析瘫痪）
4. 无答案时声明 safe default

**Pre-mortem**（Amazon 实践）—— 假想项目已失败，回溯原因，反推需明确的假设。单人 5 分钟写 3 条"最可能死因"。

### 二、工具级（当前可用）

| 工具 | 核心理念 | 适合度 |
|------|---------|--------|
| **Casper** (ThoughtWorks) | 三阶段规则流：Explore 追问 edge case → Craft TDD → Polish 独立验证 | 高，纯规则可适配 CC |
| **Claude Code Discover** (shinpr) | Vision → 假设 → Blueprint → PRD，每假设标置信度+风险维度 | 高，CC plugin |
| **ctxforge** (npm) | 检测意图 → 加载 protocol → 输出带置信度推断，等人确认后编码 | 中，协议需维护 |
| **AIxBDD** (Waterball) | 流程图 → Gherkin → 覆盖率测试计划 | 中，偏 BDD 团队 |

### 三、方法级（结构化最佳实践）

- **Example Mapping**：拿 user story，列 concrete examples（正常/边界/异常），问 AI"漏了什么"
- **Recursive Decomposition**：将目标递归分解到原子粒度，信息不足处标记 `[NEEDS CLARIFICATION]`
- **Specification by Example**：每个需求配具体场景，Gherkin 格式，BRIEF 原则写 scenario

## 与你的 SDD 工作流的结合

当前流程：`需求 → Brainstorming 门禁 → EnterPlanMode → 实现`

建议在 Brainstorming 和 EnterPlanMode 之间插入一个 **假设清单** 步骤：

1. 发给 AI 一个 Assumption Inventory prompt，产出 6+ 假设 + 置信度
2. 你扫一遍，不对的当场纠正（< 3 分钟）
3. 第一个 concrete example 写进 `.plan.md`，每个 task 追溯到一个 example
4. Code review 阶段加检查点："我当时没提到的 case，这里用了什么假设"

这三个增量不超过当前流程的 5% 时间开销，覆盖 80% 的"写完发现不对"场景。
