## claudetalk 手机端通讯架构分析

### 一、整体架构

```
手机飞书App ──┬── WS_client_A (Process A)
              ├── WS_client_B (Process B)  ← orphan
              ├── WS_client_C (Process C)  ← orphan
              └── ...  (可能 10+ 个)
```

每层内部：

```
飞书 WS 回调 → handleCardAction() → switch(action_type)
  ├─ approval-action  → handleApproachCallback(value)  → pendingApprovals Map (进程本地)
  ├─ mark-task-done   → index.json 本地标记
  ├─ execute          → callClaude()
  └─ toast/dismiss    → 立即返回
```

session 管理：

```
onMessage → clearSession()  → sessionMapCache (模块级内存)
          → /session list   → getSessionKey(convId, workDir, profile)  ← 缺 channel!
          → /session name   → getSessionKey(convId, workDir, profile)  ← 缺 channel!
          → callClaude()    → enqueueSession() → sessionQueues Map
```

### 二、审批拒绝 Bug（已确认根因）

**日志时间线**（06-22 19:23~19:26）：

```
19:23:31  Process P: registerApproval(apr_mqp4nb4l_1) → sendCard
19:24:03  飞书回调到达 → 路由到？？进程 → handleApprovalCallback
             pendingApprovals.get("apr_mqp4nb4l_1") === undefined
             → "审批请求已过期或不存在"
19:26:31  180s 超时 → "操作已取消"
```

**根因链：**

1. `ops-daemon/process_manager.py:123` — `spawn_claudetalk()` **不杀旧进程**
2. 每次重启/修复/标记文件触发新进程，旧进程还活着（日志 06-20~06-22 共 **16 次** `startBot`）
3. 每个进程各有一个独立的飞书 WS 长连接，各有自己的 `pendingApprovals` Map
4. Process P（最后启动的）注册审批 → 存到 P 的 Map
5. 飞书下推 `card.action.trigger` → **随机选一个 WS 连接** → 如果选中了旧进程 A/B/C...
6. 旧进程的 Map 是空的 → `pendingApprovals.get()` → `undefined` → **"审批请求已过期或不存在"**
7. Process P 的 `resolve()` 永远等不到 → 180s 超时

**对比验证：** `spawn_proxy()` 和 `spawn_mcp_server()` 都有 `_kill_port()` 前置杀旧进程，唯独 `spawn_claudetalk()` 没有。

### 三、Session 管理的 Bug / 逻辑冲突

#### Bug 1：`getSessionKey` 参数不一致 — session 操作前后矛盾

**`src/index.ts`（/new 指令，正常路径）：**
```typescript
clearSession(conversationId, workDir, profile)  
// 内部 getSessionKey → convId\x00workDir\x00profile\x00channelType
```

**`src/commands/session.ts`（/session list/new/name/status）：**
```typescript
getSessionKey(conversationId, workDir, profile)
// 内部 getSessionKey → convId\x00workDir\x00profile\x00undefined
```

**后果：** `/new` 清除的是 4 段 key 的 session，`/session status` 读的是 3 段 key 的结果。它们指向**不同的 Map entry**。所以 `/session status` 可能显示"当前没有活跃会话"，但 `/new` 确实能用。

#### Bug 2：多进程 `sessionMap` 覆盖

`claude.ts` 的 `sessionMapCache` 是进程本地内存。每个进程独立加载 → 修改 → 写回 `claudetalk-sessions.json`。多进程共存时，后写回的会覆盖前面的进度。

#### Bug 3：`clearedDuringProcessing` 窗口期

`clearSession()` 在 claude CLI 进程已启动后才调用时，claude 返回的 `session_id` 可能被重新写回，尽管有 `clearedDuringProcessing` 防护，但 `compactSession()` **不检查这个标记**：
- 压缩完成后会把 sessionId 写回 Map
- 如果用户刚刚 `clearSession()` 且在 compact 完成之前，旧的 sessionId 会重新出现

#### Bug 4：`sessionQueues` 异常链

`enqueueSession()` 中的 `next.then(() => {})` 吞了 reject。如果 `fn()` 抛出异常，后续排队的消息永久卡在 pending 队列中：

```typescript
// index.ts:678 — 当前代码
const next = prev.then(() => fn())
sessionQueues.set(sessionKey, next.then(() => {}))  // ← fn() 的 reject 被 .then(() => {}) 吞了
// 链断裂，后续消息永远 pending
```

### 四、其他架构问题

#### 问题 1：`stop()` 不关闭 WS 连接

`index_feishu.ts:631` — 只 `wsClient = null`，**没有调用 SDK 的 close 方法**，底层 WS 连接泄漏。

#### 问题 2：Peer message TOCTOU 竞态

多个 bot 实例轮询同一个 `bot_{name}.json`，没有文件锁，读→改→写窗口期数据丢失。

#### 问题 3：`processedEventIds` O(n) 退化

每收到一条消息就遍历整个 Map 做过期清理，长期运行 Map 可能积累数万条。

#### 问题 4：权限验证靠客户端字段

`handleApprovalCallback` 的 `clickerId` 来自卡片回调 payload，`requesterId` 来自卡片构建时的 payload，两者都是**客户端可控**。虽然飞书签名机制有一定保护，但逻辑上不应信任客户端身份。

---

### 五、修复优先级建议

| 优先级 | 问题 | 修复 |
|--------|------|------|
| **P0** | 审批回调路由到 orphan 进程 | `spawn_claudetalk()` 前加 `_kill_by_pid_file()`，对齐 proxy/mcp |
| **P1** | `getSessionKey` 参数缺 channel | `session.ts` 所有调用补上 `channel` 参数 |
| **P1** | `sessionQueues` 异常链断裂 | `next.then(() => {}, () => {})` 或加 catch |
| **P2** | `compactSession` 不检查 cleared 标记 | compact 前检查 `wasClearedDuringProcessing` |
| **P2** | `stop()` 不关 WS | 调用 SDK 的 close |
| **P3** | Peer message TOCTOU | 加文件锁或改为 IPC 通信 |
| **P3** | `processedEventIds` O(n) | 改为定时全清或用 WeakRef |

---

### 结论

**审批失败是 `spawn_claudetalk()` 的 P0 bug 导致的多实例副作用，不是审批逻辑本身的 bug。** Session 管理有两个代码路径的 key 不一致问题，但实际使用中因为大多数用户用同一条 channel，channel 为 `undefined` 时仍可能"碰巧"匹配同一个 entry，影响有限。最紧急的修复是杀掉旧进程再启动——和其他子进程的管理方式对齐。
