你问的是 GitHub 上开源的 **CodeGraph** 类项目群（有好几个同名或类似项目）。我来对比一下它们和你已有的 `memory-mcp` 的 `code-graph` 能力。

---

## 你已有的

`memory-mcp` 的 `code-graph` 已经覆盖了：
- **25 个项目的跨项目运行时依赖**（HTTP 调用、subprocess spawn、文件系统读取）
- **BFS 影响链追踪**（`memory_trace_impact`，深度可配）
- **符号查询**（`memory_code_query`：定义位置、消费者、导出者）

这是**运行时/项目边界级**的依赖图谱，适合回答"改了这个文件的接口，哪些项目会挂"这类问题。

---

## GitHub 社区项目能补充什么

有几个方向值得关注：

### 1. xnuinside/codegraph — Python 静态依赖图
- 纯 Python 静态分析，pip 安装即用
- 生成类/模块/方法间的 **import 依赖图**（matplotlib/networkX 可视化）
- `pip install codegraph`
- **价值**：补你的空白——你现有的是跨项目运行时依赖，缺少**单项目内部**的函数/类级静态依赖。如果你在做 Python 项目内重构，它有用
- **但**：只支持 Python，且长期未大更新，star 300+，不算活跃

### 2. codegraph-cli — 多语言 + AI 语义分析
- 基于 tree-sitter，支持多语言
- impact analysis、语义搜索、dead code 检测、CrewAI 多 agent 代码生成
- **价值**：功能集比你现有的更丰富（尤其是 dead code 检测和语义搜索），但体量太大——装一套 crewAI 管线就重了
- **对于你的场景**：过重了。你不需要 AI 代码生成和 RAG chat

### 3. The-Code-Graph — Rust 实现，轻量快速
- Rust 写的 CLI（`tcg`）：index / find / refs / impact / dead-code
- 多语言支持（tree-sitter）
- **价值**：如果需要在 CC 之外独立查询代码依赖（不走 memory-mcp），这个最快
- **但**：需要 `cargo install`，且和 memory-mcp 的能力有大量重叠

---

## 结论

**目前帮助有限，不推荐引入。**

原因：
1. 你的 `memory-mcp` 的 `code-graph` 已经覆盖了最高价值的场景——**跨项目变更影响链**，这是社区项目做不到的（它们只分析单项目）
2. 社区项目能补的**单项目内静态依赖**，你的使用场景里需求不强——你的项目大多在 1-2 个文件级别修改，不需要全项目函数级依赖图
3. 多一个工具 = 多一层维护 + 扫描开销

**如果将来需要更细粒度的单项目分析**，可以单独评估 `xnuinside/codegraph` 或 `The-Code-Graph`，但目前没有收益。
