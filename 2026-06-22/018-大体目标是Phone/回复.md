## 方案 A：文件级解耦（轻量，当前模式延伸）

```
D:\ClaudeProjects\tasks/
├── index.json          ← 唯一数据源，拓展 schema
│   {task-id}: {type, status, summary, source, actions, ...}
├── cli.py              ← CLI：list --type / mark-done / create / status
├── tasks-mcp.ts        ← MCP Server（可选），暴露 task_create / task_list / mark_done
└── producers/          ← 选配：扫描各项目产出灌入 index.json
```

**推模式**：ops-daemon 的 Python 代码直接 `tasks/cli.py create --type tool-recommend ...`
**拉模式**：`tasks/cli.py sync --from ../ops-daemon/data/working/tool_audit_cache.json`

Claudetalk 仍直接读 `index.json`，只是路径改为 `../tasks/index.json`。

**优点**：改动量最小，现有 mark-task-done callback 几乎不改
**缺点**：Claudetalk 对 tasks 有直接文件依赖，非真正"独立"

## 方案 B：MCP 服务独立（中等）

```
D:\ClaudeProjects\tasks/
├── src/tasks-server.ts     ← MCP Server，管理全部任务
├── index.json / tasks.db   ← 存储，仅 MCP Server 读写
├── cli.ts                  ← CLI 包装 MCP 调用
└── tasks-mcp.skill         ← Claude Code Skill，终端入口
```

所有读写通过 MCP 接口：
- `tasks_create` → 任意项目（ops-daemon / CC 会话）创建任务
- `tasks_list` → 按 type/status/source 过滤
- `tasks_mark_done` → 标记完成（含 callbacks）
- `tasks_process` → 交互式处理（审核类任务逐条操作）

Claudetalk 通过 MCP client 调用 tasks list，不再直接读文件。飞书卡片 `mark-task-done` callback 调用 MCP 接口写回。

**优点**：真正独立，接口契约化，任意项目都可接入
**缺点**：需要建 TS MCP Server，Claudetalk 需要 MCP client 集成（已有基础）

## 方案 C：事件驱动（完善但重）

B 的基础上增加事件总线：任务创建/完成/拒绝时广播事件，订阅方（Claudetalk / CC）自动收到通知。适合实时协作场景，但对单用户系统 overkill。
