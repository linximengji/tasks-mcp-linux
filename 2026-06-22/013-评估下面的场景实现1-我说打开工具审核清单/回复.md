基于已有的代码，评估两个场景的可实现程度：

---

## 场景 1：工具审核清单（交互式）

**现在能做到的：**
- 按需触发 `tool_audit.py`（不等周一），输出未注册工具 + L1/L2/L3 推荐分级
- 读 `tool_audit_rejected.json` 过滤已拒绝项
- 展示清单（在 CC 对话中或发飞书卡片）

**各 action 实现难度：**

| Action | 难度 | 现状 |
|--------|------|------|
| 加入拒绝清单 | 低 | `tool_audit_rejected.json` 已有格式，直接 append |
| 暂时保留 | 低 | 跳过即可，下次审计还会出现 |
| 安装成功 | **高** | 需要：探测工具命令/路径 → 生成 `.mcp.json` 配置片段 → 注册 → 验证连通。`mcp-builder` skill 能指导生成，但全自动安装仍需工具发现逻辑 |

**缺口：** 目前没有"按需触发 + 展示 + 逐条处理"的 skill。审计逻辑是 batch 模式（跑一次出报告），不是 API 模式（查一次返回待处理队列）。

---

## 场景 2：记忆审核清单（交互式）

**现在能做到的：**
- `memory_cleanup action=list` MCP 工具已经返回低质量候选（confidence < 0.4）
- `memory_review action=apply/dismiss` MCP 工具已经能应用或驳回 `review-report.md` 里的建议
- L1/L2/L3 审计已有完整逻辑，会产出 `review-report.md`

**各 action 实现难度：**

| Action | 难度 | 现状 |
|--------|------|------|
| 列出待人工处理 | 低 | `memory_cleanup list` + `review-report.md` 的 pending 项 |
| 确认删除（L1） | 低 | `memory_cleanup action=delete` 已实现 |
| 接受修正建议 | 低 | `memory_review action=apply` 已实现 |
| 驳回建议 | 低 | `memory_review action=dismiss` 已实现 |
| 补充缺失记忆 | 中 | L3 会列出 missing segments，但没有"一键创建空模板"工具 |

**缺口：** MCP 工具层都齐了，缺的是上层 skill 把这些工具串成一个"打开 → 展示 → 逐条处理 → 完成"的交互流。

---

## 总结

```
场景1（工具审核）: 展示+拒绝+保留 ≈ 70% 可实现，自动安装 ≈ 40%
场景2（记忆审核）: 全流程 ≈ 85% 可实现，MCP 工具已就绪，缺 skill 串联
```

**场景 2 更成熟**——底层工具都有，写一个 skill（类似 `feishu-remote` 的模式）串起来就能跑。

**场景 1 的瓶颈是"安装"**——工具发现没有通用方案（每个工具的 CLI 入口、依赖、配置方式都不同），能做到"推荐安装 → 人工确认后辅助配置"已经不错，全自动安装不现实。

要不要先实现场景 2 的 skill？它的 ROI 更高，能立刻用起来。
